# Migrating from Tulpar UI v0.6.x to v0.10

v0.10 is a purely additive release. There are **no breaking changes** to any
existing component — the button family (v0.1–v0.4), the input family (v0.5),
or the shell family (v0.6). Updating to v0.10 requires no migration work if you
only use existing components; everything below is new functionality you can
adopt at your own pace.

See the full design spec: [`docs/superpowers/specs/2026-06-17-tulpar-ui-v0.10-selection-family-design.md`](../superpowers/specs/2026-06-17-tulpar-ui-v0.10-selection-family-design.md)

---

## What's new — five public custom elements

### `<tulpar-switch>`

An immediate-apply on/off toggle. A switch commits its change on interaction
(no pending-submit semantic, unlike checkboxes). Form-associated: submits its
`value` (`"on"` by default) under `name` when checked.

Key differentiators:
- `loading` — async-pending state: spinner in thumb, toggle suppressed, `aria-busy`. Use it when the toggle triggers a server call before it should commit.
- `show-icon` — reveals built-in check/cross glyphs in the thumb (hidden by default).
- `slot="icon-on"` / `slot="icon-off"` — custom SVG icons inside the thumb (e.g., sun/moon for a theme toggle).
- `on-color` / `off-color` — per-instance track color overrides.

### `<tulpar-checkbox>`

A staged boolean checkbox. Supports checked, unchecked, and `indeterminate`
(presentational/programmatic only — clicking an indeterminate checkbox resolves
to `checked`, per Spectrum behavior). Form-associated: submits `value` under
`name` while checked.

Key features:
- `indeterminate` — renders the dash glyph and sets `aria-checked="mixed"`. Does NOT change `checked`.
- `slot="icon"` — replaces the built-in tick with a custom SVG.
- `variant="card"` — wraps in a bordered card surface for option pickers.

### `<tulpar-radio>`

A single radio button, meaningful within a `<tulpar-radio-group>`. Selecting it
sets `checked=true`; a radio never toggles itself off. Form-associated: submits
its `value` under the group's `name` while checked.

- `variant="card"` — card surface, same as checkbox.
- Standalone use is supported; group orchestration adds arrow-key navigation.

### `<tulpar-radio-group>`

A single-select group of `<tulpar-radio>`. Not form-associated itself — the
checked child submits natively. Implements the WAI-ARIA radiogroup pattern:
roving tabindex, arrow-key selection-follows-focus, Home/End.

- `value` — the `value` of the currently checked radio, or `null`.
- `orientation="vertical | horizontal"` — controls flex layout; default `vertical`.
- Propagates `name`, `size`, `disabled`, `readonly`, `color` to children.

### `<tulpar-checkbox-group>`

A multi-select group of `<tulpar-checkbox>`. Not form-associated — each checked
child submits its own name/value pair (multi-entry). Children keep independent
Tab/Space focus.

- `value` — `string[]` of the checked children's values, in DOM order.
- Emits `CustomEvent('change', { detail: { value: string[] } })`.
- Propagates `name`, `size`, `disabled`, `readonly`, `color` to children.

## Internal bases (not public API)

Two unexported abstract bases were added to `@tulpar-ui/core/_internal`:

- **`SelectionControlBase`** — shared by Switch, Checkbox, Radio. Owns form-association, the inline anatomy (label + description + message row), sizing, state, ARIA wiring.
- **`SelectionGroupBase`** — shared by RadioGroup, CheckboxGroup. Owns orientation layout, group-level label/legend, propagation, and the group message row.

These are implementation details. They are never exported from `@tulpar-ui/core`
and their type is never part of the public API.

## Wrappers

| Framework | Switch | Checkbox | Radio | RadioGroup | CheckboxGroup |
|---|---|---|---|---|---|
| Angular | `<tulpar-switch-ng>` | `<tulpar-checkbox-ng>` | `<tulpar-radio-ng>` | `<tulpar-radio-group-ng>` | `<tulpar-checkbox-group-ng>` |
| Vue | `<TulparSwitch>` | `<TulparCheckbox>` | `<TulparRadio>` | `<TulparRadioGroup>` | `<TulparCheckboxGroup>` |

Angular wrappers are signal-based (`input()` / `output()`). Vue wrappers support
`v-model` for `checked` (boolean) on Switch/Checkbox and `modelValue` on the
groups.

## Shared anatomy

All five elements share the following anatomy, implemented in the (unexported)
internal `SelectionControlBase` / `SelectionGroupBase`.

### Label

Provide via attribute or slot — slot wins when both are present:

```html
<!-- attribute -->
<tulpar-switch label="Enable notifications"></tulpar-switch>

<!-- slot wins over attribute -->
<tulpar-checkbox label="fallback">
  <span slot="label">I agree to the <a href="/terms">Terms of Service</a></span>
</tulpar-checkbox>
```

### Label position

`label-position` accepts `start | end` (default `end`).

```html
<tulpar-switch label="Dark mode" label-position="start" checked></tulpar-switch>
```

### Description slot

An optional secondary line beneath the label:

```html
<tulpar-switch checked>
  <span slot="label">Weekly digest</span>
  <span slot="description">Sent every Monday at 9 AM with your activity summary.</span>
</tulpar-switch>
```

### Message row

A single always-reserved row below the control displays helper, error, or
warning text. Priority: `error-text` > `warn-text` > `helper-text`. Remove the
reserved space with `no-message-space`.

```html
<tulpar-checkbox
  label="Accept terms"
  required
  invalid
  error-text="You must accept to continue"
></tulpar-checkbox>
```

### Status taxonomy

| Attribute | Meaning |
|---|---|
| `invalid` | Validation failed — shows `error-text`, error icon |
| `warn` | Soft warning — shows `warn-text`, warning icon |

### Size

`size` accepts `xs | sm | md | lg | xl` (default `md`). Matches the input family
for cross-component alignment.

### Color override

`color` overrides the fill/track color per instance or per group without
touching design tokens:

```html
<tulpar-checkbox color="#ef4444" checked label="Danger"></tulpar-checkbox>
<tulpar-switch on-color="#10b981" label="Custom on-color" checked></tulpar-switch>
```

## Quick-start examples

### Settings form (switches)

```html
<form>
  <tulpar-switch
    name="email-notifications"
    label="Email notifications"
    checked
    helper-text="At most 3 emails per day"
  ></tulpar-switch>

  <tulpar-switch
    name="push-notifications"
    label="Push notifications"
  ></tulpar-switch>

  <tulpar-switch
    name="weekly-digest"
    label="Weekly digest"
    checked
  ></tulpar-switch>
</form>
```

### Terms acceptance (required switch + checkbox)

```html
<tulpar-switch
  name="tos"
  value="accepted"
  label="I accept the Terms of Service"
  required
  invalid
  error-text="You must accept to continue"
></tulpar-switch>

<tulpar-checkbox
  name="privacy"
  value="accepted"
  required
>
  <span slot="label">I have read the <a href="/privacy">Privacy Policy</a></span>
</tulpar-checkbox>
```

### Async toggle (switch loading)

```html
<!-- Core Web Component -->
<tulpar-switch id="auto-save-toggle" label="Auto-save" name="auto-save"></tulpar-switch>

<script>
  const sw = document.querySelector('#auto-save-toggle');
  sw.addEventListener('change', async () => {
    const intended = sw.checked;
    sw.checked = !intended; // revert optimistic toggle
    sw.loading = true;
    try {
      await fetch('/api/settings/auto-save', {
        method: 'POST',
        body: JSON.stringify({ enabled: intended }),
        headers: { 'Content-Type': 'application/json' },
      });
      sw.checked = intended; // commit after success
    } catch {
      // leave reverted — user can retry
    } finally {
      sw.loading = false;
    }
  });
</script>
```

### Multi-select filters with select-all (CheckboxGroup)

```html
<tulpar-checkbox
  id="select-all"
  label="All statuses"
></tulpar-checkbox>

<tulpar-checkbox-group id="status-group" name="status">
  <tulpar-checkbox value="active" label="Active" checked></tulpar-checkbox>
  <tulpar-checkbox value="pending" label="Pending" checked></tulpar-checkbox>
  <tulpar-checkbox value="archived" label="Archived"></tulpar-checkbox>
</tulpar-checkbox-group>

<script>
  const parent = document.querySelector('#select-all');
  const group = document.querySelector('#status-group');

  const syncParent = (values) => {
    const total = group.querySelectorAll('tulpar-checkbox').length;
    parent.checked = values.length === total;
    parent.indeterminate = values.length > 0 && values.length < total;
  };

  parent.addEventListener('change', () => {
    const checked = parent.checked;
    group.querySelectorAll('tulpar-checkbox').forEach(cb => cb.checked = checked);
    group.value = checked
      ? Array.from(group.querySelectorAll('tulpar-checkbox')).map(cb => cb.value)
      : [];
  });

  group.addEventListener('change', (e) => syncParent(e.detail.value));

  // Seed on load.
  syncParent(group.value);
</script>
```

### Vue — two-way binding

```vue
<script setup lang="ts">
import { ref } from "vue";
import { TulparSwitch, TulparCheckboxGroup, TulparCheckbox } from "@tulpar-ui/vue";

const darkMode = ref(false);
const selectedChannels = ref<string[]>(["email"]);
</script>

<template>
  <TulparSwitch v-model="darkMode" label="Dark mode" />

  <TulparCheckboxGroup v-model="selectedChannels" label="Notify via">
    <TulparCheckbox value="email" label="Email" />
    <TulparCheckbox value="push" label="Push" />
    <TulparCheckbox value="sms" label="SMS" />
  </TulparCheckboxGroup>
</template>
```

### Angular — signal-based inputs

```ts
import { Component, signal } from '@angular/core';
import { TulparSwitchNg, TulparRadioGroupNg, TulparRadioNg } from '@tulpar-ui/angular';

@Component({
  standalone: true,
  imports: [TulparSwitchNg, TulparRadioGroupNg, TulparRadioNg],
  template: `
    <tulpar-switch-ng
      label="Enable feature flags"
      [checked]="featureFlags()"
      (change)="featureFlags.set($event.target.checked)"
    />

    <tulpar-radio-group-ng
      label="Billing cycle"
      [value]="billing()"
      (change)="billing.set($event.detail.value)"
    >
      <tulpar-radio-ng value="monthly" label="Monthly" />
      <tulpar-radio-ng value="annual" label="Annual" />
    </tulpar-radio-group-ng>
  `,
})
export class SettingsComponent {
  featureFlags = signal(false);
  billing = signal<string | null>('monthly');
}
```

## Deferred / out of scope

The following were considered for v0.10 and explicitly deferred:

- **Track-internal on/off text** (Ant `checkedChildren`/`unCheckedChildren`, Carbon small-toggle state text). Thumb icons are supported instead.
- **`max-selected` on CheckboxGroup** — consumer-enforced in the `change` handler.
- **Built-in "select all" in CheckboxGroup** — the `indeterminate` API + the recipe above (~12 lines) is intentionally left to the consumer.
- **Segmented control / toggle-button-group** — belongs to the button family; deferred.
- **React wrapper.**
