# Migrating from Tulpar UI v0.3.x to v0.4

v0.4 is a hardening release: 6 bug fixes and 14 enterprise-grade improvements.
Because Tulpar UI is pre-1.0, this release ships behavior changes alongside
the additive improvements. This guide walks every observable change.

## Peer dependency bumps

| Package | Old | New |
|---|---|---|
| `@angular/core` | `^20.0.0` | `^22.0.0` |
| `@angular/common` | `^20.0.0` | `^22.0.0` |
| `vue` | `^3.4.0` | `^3.5.0` |
| `lucide-angular` | `>=0.400.0` | `>=0.500.0` |
| `zone.js` | (dev) | removed — Angular 22 zoneless default |

Browser baseline: Chromium 65+, Firefox 78+, **Safari 17+**.

## Behavior changes

### Form submission

`<tulpar-button type="submit">` no longer dispatches duplicate submit events.
If your app relied on the old behavior, update your handler to expect exactly
one submit per click.

`name` and `value` are now reflected to the parent `<form>` on submission,
matching native `<button>`. Apps that added hidden inputs to work around this
should remove those workarounds (otherwise FormData will contain duplicate
keys).

**Semantic caveat:** Chromium does not accept form-associated custom elements
as the `submitter` argument to `form.requestSubmit()`. v0.4 pushes the
button's value into FormData via `ElementInternals.setFormValue()` only
during the click handler (just before `requestSubmit()`) and clears it
immediately after. This produces native-equivalent behavior in the common
case (one button click → its name/value in FormData). It differs from native
semantics if two `<tulpar-button>`s in the same form race a programmatic
submit — only the most recent click's value lingers briefly.

### Disabled anchor

`<tulpar-button href="…" disabled>` no longer sets `href="javascript:void(0)"`.
The anchor remains inert via `aria-disabled="true"` and `tabindex="-1"`. CSP
configurations that blocked `javascript:` URLs no longer trigger violations.

### ButtonGroup ARIA role

`<tulpar-button-group>` host now exposes `role="toolbar"` (was `group`) with
`aria-orientation="horizontal"` (or `vertical` when stacked). Screen reader
users hear "toolbar" instead of "group".

Apps that queried `[role="group"]` to find button groups must update to
`[role="toolbar"]`. Note: the role is set via `ElementInternals.role`, not as
a DOM attribute, so `getAttribute("role")` returns `null` — query by element
tag (`tulpar-button-group`) or by accessible name.

### Focus ring

Focus indication switched from `box-shadow` to `outline`. Visual position is
similar (2px outside the button). Apps that disabled the focus ring via
`box-shadow: none` overrides should use `outline: none` instead.

The ring now responds to `forced-colors: active` (Windows High Contrast Mode),
where it adopts `CanvasText` for system contrast.

### Console output

Development-mode warnings unchanged. Production bundles no longer log button
ARIA validation warnings (a11y checks happen in dev only).

### Wrapper DOM (Angular)

`<tulpar-button-ng>` host now uses `display: contents`. The wrapper element
is excluded from layout, becoming transparent to the box tree. CSS that
targets `tulpar-button-ng { ... }` for layout (width, padding, border) will
not take effect — apply layout to `<tulpar-button>` or wrap externally.

### Safari 16

`delegatesFocus` on form-associated custom elements is unreliable in Safari
≤ 16. Tulpar UI v0.4 baselines Safari 17+. On Safari 16, focus still reaches
the inner button, but the focus ring on that inner button may not render.

### Keyframe namespace

The spinner animation keyframe was renamed `tulpar-spin` → `tulpar-button-spin`.
Apps that referenced `tulpar-spin` in custom animations (uncommon — it was
shadow-scoped) must update.

### ButtonGroup dynamic children

ButtonGroup now uses a `MutationObserver` and re-syncs `tabindex` +
`data-group-position` whenever children are inserted, removed, or reordered.
Apps that previously had to call a refresh method manually can drop that
code.

## Type imports

The wrapper packages no longer redefine the type unions. Consumers can keep
importing from the wrapper packages (they re-export from core) or switch to:
```ts
import type { ButtonSeverity, ButtonVariant } from "@tulpar-ui/core";
```

## New props on `tulpar-button`

| Prop | Type | Purpose |
|---|---|---|
| `name` | string | Form submission key (matches native `<button>`) |
| `value` | string | Form submission value (matches native `<button>`) |

## Angular zoneless

The Angular wrapper now targets Angular 22 with zoneless change detection.
Apps still bootstrapping with `provideZoneChangeDetection()` continue to
work — signals don't require zoneless. To opt in:

```ts
import { provideZonelessChangeDetection } from "@angular/core";

bootstrapApplication(AppComponent, {
  providers: [provideZonelessChangeDetection()],
});
```

`zone.js` can be removed from `polyfills` and `devDependencies` once
all upstream code is zoneless-safe.

## Tooling notes

### Angular tests

The wrapper now uses `@analogjs/vitest-angular` for TestBed-backed Vitest
tests. Consumer apps are unaffected — this only changes how tulpar-ui itself
is tested.

### Token-derived fallbacks

`@tulpar-ui/tokens/fallbacks` is a new sub-path export that emits a TypeScript
constant of fallback color values derived from primitive tokens. Used by
`@tulpar-ui/core` to keep CSS `var(--…, fallback)` chains in sync with token
primitives. No action needed for consumers.
