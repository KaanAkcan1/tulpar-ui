# Tulpar UI v0.3 — Button API Expansion

**Status:** Approved 2026-06-08
**Predecessor:** [v0.2 — Design system redesign](./2026-06-08-tulpar-ui-v0.2-design-system-redesign.md)
**Owner:** Kaan Akcan

---

## 1. Overview

v0.3 replaces v0.2's single-axis `variant` enum with a PrimeNG-style multi-axis Button API: orthogonal `severity` (semantic intent) + `variant` (visual fill) + composable modifiers (`raised`, `block`, `justify`, `shape`) + a per-instance `color` override that unlocks all 21 chromatic families from the v0.2 palette. Adds loading-label / loading-position, custom spinner slot, `data-disabled` passthrough, and stacked ButtonGroup layout.

**Scope of change:**
- `@tulpar-ui/tokens` package: add 3 new semantic variants (`info`, `help`, `contrast`); rename existing `warning` → `warn` for parity with severity prop value.
- `@tulpar-ui/core` Button + ButtonGroup: major API expansion, new render branches for variants, JS-driven color override, new slots.
- `@tulpar-ui/angular` wrapper: new signal-based inputs matching the core API.
- `@tulpar-ui/vue` wrapper: new props matching the core API.
- Storybook + playgrounds: updated stories + demo pages to exercise the matrix.

**Inspiration & research credits (synthesized from top-5 enterprise libraries):**
- PrimeNG/PrimeVue — `severity` enum, modifier booleans baseline
- Mantine v7 — `tonal` variant value (their `light`), `justify` prop, `data-disabled` semantics
- Ant Design v5 — `shape: round | circle` separation
- Chakra UI v3 — `loadingText` (we use `loading-label`)
- MUI v6 — `loadingPosition`
- IBM Carbon — stacked ButtonSet, icon-only a11y enforcement

## 2. Goals & Non-Goals

**Goals:**
- Match enterprise depth: 8 severities × 5 variants × 4 modifier dimensions, all compose orthogonally.
- `color` prop unlocks the gold-button and any chromatic-family use case (`<tulpar-button color="emerald">`) without forking the component.
- Make icon-only buttons trivial and accessible by default (warn on missing label).
- Loading state with the three common patterns (`start` spinner, `center` overlay, `end` spinner) plus custom-icon slot.
- `data-disabled` correctness for tooltip-wrapped and `<a>`-polymorphic buttons.

**Non-Goals (binding):**
- No `gradient` variant — incompatible with financial-grade aesthetic.
- No `ghost` boolean modifier (Ant pattern); `ghost` stays as a `variant` value.
- No ripple / `disableRipple` — Tulpar doesn't use ripple animations.
- No `autoInsertSpace` / CJK-specific layout.
- No `autoContrast` runtime calculation — tokens enforce contrast.
- No built-in tooltip — out of v0.3 scope; consumers compose a separate `<tulpar-tooltip>` (future).
- No `component` / `as` polymorphism on the core WC — `href` handles the anchor case; framework wrappers are not getting an `as` prop.
- No granular `radius=xs|sm|md|...` prop — `shape` is enough for common cases.
- No new compound components: Split Button, Toggle Button, FAB still out (per v0.1 spec).
- No skeleton state — separate concern for a future Skeleton component.
- **No bundled icon set.** Tulpar does not ship Lucide, Heroicons, or any other icons. Consumers BYO. Sizing normalization is provided via `::slotted()` + the `--tulpar-button-icon-size` token (see §9). A separate `@tulpar-ui/icons-lucide` adapter package may ship later but is out of v0.3.

## 3. API Surface

### Attributes (Web Component)

| Attribute | Type | Default | Notes |
|---|---|---|---|
| `severity` | enum (see §4) | `"primary"` | Semantic intent — chooses the chromatic family |
| `variant` | enum (see §5) | `"solid"` | Visual fill style |
| `color` | enum (see §6) | unset | Override the chromatic family — bypasses severity's color mapping but keeps everything else (sizes, variant logic) intact |
| `shape` | `"default" \| "round" \| "circle"` | `"default"` | Border radius + aspect ratio policy |
| `size` | `"xs" \| "sm" \| "md" \| "lg" \| "xl" \| "2xl" \| "3xl"` | `"md"` | Same scale as v0.2 |
| `raised` | boolean | `false` | Adds `shadow.md` regardless of variant |
| `block` | boolean | `false` | Full-width (`width: 100%`) |
| `justify` | `"start" \| "center" \| "end" \| "between"` | `"center"` | Children alignment, especially with `block` + icons |
| `disabled` | boolean | `false` | Native `disabled` on button / `aria-disabled` + `tabindex=-1` on anchor |
| `data-disabled` | boolean (attribute presence) | absent | Style-only disabled — does NOT block pointer events. For tooltip-wrapped or anchor-polymorphic buttons. |
| `loading` | boolean | `false` | Active loading state — suppresses click + shows spinner |
| `loading-label` | string | unset | Replaces the default slot label during loading |
| `loading-position` | `"start" \| "center" \| "end"` | `"center"` | Spinner placement during loading |
| `icon-only` | boolean | `false` | Square padding mode; requires `aria-label` (warns) |
| `icon-position` | `"start" \| "end" \| "top" \| "bottom"` | `"start"` | Flex direction for icon + label layout (see §9) |
| `icon-separator` | boolean | `false` | 1px divider between icon area and label area (see §9) |
| `href` | string | unset | Renders `<a>` instead of `<button>` |
| `target` | string | unset | Anchor target |
| `rel` | string | unset | Anchor rel |
| `type` | `"button" \| "submit" \| "reset"` | `"button"` | Form integration via `ElementInternals` |
| `aria-label` | string | unset | Required for `icon-only` / `shape="circle"` |

### Slots

| Slot | Purpose |
|---|---|
| default | Button label |
| `start` | Leading icon |
| `end` | Trailing icon |
| `loading-icon` | Custom spinner (replaces default `.spinner`) |

### CSS custom properties

All v0.2 `--tulpar-button-*` properties remain valid for consumer-level overrides. v0.3 adds:
- `--tulpar-button-shadow-raised` — shadow used when `raised` is true (default: `var(--tulpar-shadow-md)`)
- `--tulpar-button-radius-round` — radius for `shape="round"` (default: `9999px`)
- `--tulpar-button-icon-size` — icon dimension applied to slotted children via `::slotted()`. Per size: 12/14/16/18/20/24/28px for xs/sm/md/lg/xl/2xl/3xl. Consumer can override per-instance.
- `--tulpar-button-separator-color` — divider color for `icon-separator`. Default resolves per variant (border-default for outlined/tonal/ghost; rgba(255,255,255,0.2) for solid).

## 4. Severity → chromatic family mapping

The semantic layer adds three new `VariantColorTokens` slots (`info`, `help`, `contrast`) and renames `warning` → `warn`. The full mapping table for `tulparLight`:

| Severity | Family | 600 (default) | 700 (hover) | 800 (active) | onColor |
|---|---|---|---|---|---|
| `primary` | navy | `#133a66` | `#0a2540` | `#061a30` | `stone.50` (#fafaf9) |
| `secondary` | stone | `#57534e` | `#44403c` | `#292524` | `stone.50` |
| `info` | blue | `#2563eb` | `#1d4ed8` | `#1e40af` | `stone.50` |
| `success` | green | `#16a34a` | `#15803d` | `#166534` | `stone.50` |
| `warn` | amber | `#d97706` | `#b45309` | `#92400e` | `stone.50` |
| `help` | purple | `#9333ea` | `#7e22ce` | `#6b21a8` | `stone.50` |
| `danger` | red | `#dc2626` | `#b91c1c` | `#991b1b` | `stone.50` |
| `contrast` | stone (inverted) | `stone.900` (#1c1917) | `stone.800` (#292524) | `stone.700` (#44403c) | `stone.50` |

> **Note on `primary`:** v0.2 mapped `color.brand.default` to `navy.700`. v0.3 shifts the *default* to `navy.600` because deeper-on-light is more standard across libraries. Hover deepens to `navy.700`. This is a slight tone shift; spec calls it out for tracking.

For `tulparDark`, all severities use the 400 stop as default (lighter for contrast against dark surfaces) and 300/200 for hover/active. `contrast` inverts: uses `stone.50` as default for the dark theme.

## 5. Variant → visual style logic

Each variant defines four state colors derived from the chosen chromatic family (`F` = severity's family or `color` override family).

| Variant | Default bg | Default text | Default border | Hover behavior |
|---|---|---|---|---|
| `solid` | `F.600` | `F-onColor` (auto: stone.50) | transparent | bg → `F.700` |
| `outlined` | transparent | `F.700` | `F.600` | bg → `F.50`, border → `F.700` |
| `tonal` | `F.50` | `F.700` | transparent | bg → `F.100` |
| `ghost` | transparent | `F.700` | transparent | bg → `F.50` |
| `link` | transparent | `F.700` | none | underline + bg unchanged |

**`link` is special:** ignores `raised`, `shape`, `block`, `size` height (renders inline). All other variants honor all modifiers.

**Dark mode**: each variant uses the corresponding lighter stops (default: 400, hover: 300, etc.) per the v0.2 dark theme convention.

## 6. Color override

When the `color` attribute is set, the component overrides the chromatic family entirely. `severity` has no further visual effect in that case — it remains reflected to the host element as a data attribute (`severity="danger"` shows up in DOM) so consumer-side CSS and analytics can still read intent, but the rendered colors come exclusively from the `color` value. The variant logic still applies, just sourced from the new family.

Accepted values (21 families from v0.2 primitives):

```
navy, gold, stone, slate,
red, orange, amber, yellow, lime, green, emerald, teal,
cyan, sky, blue, indigo, violet, purple, fuchsia, pink, rose
```

**Implementation pattern:** The component reads the `color` attribute and JS-sets six CSS custom properties on the host element:

```js
const stops = { default: 600, hover: 700, active: 800, subtle: 50, subtleHover: 100, border: 600 };
for (const [role, stop] of Object.entries(stops)) {
  this.style.setProperty(`--_btn-${role}`, `var(--tulpar-primitive-color-${color}-${stop})`);
}
```

The CSS rules read from these `--_btn-*` private custom properties. No `color × variant × state` CSS combinatorial explosion. Dark mode picks lighter stops via a parallel set when `.dark` class is on `<html>`.

**Examples:**
- `<tulpar-button color="gold">Upgrade</tulpar-button>` — gold solid button
- `<tulpar-button color="emerald" variant="outlined">Approve</tulpar-button>` — emerald outlined
- `<tulpar-button severity="danger" color="rose">Delete</tulpar-button>` — rose color but danger semantic (rare; documented for completeness)

## 7. Shape

| Shape | Border radius | Aspect ratio | Notes |
|---|---|---|---|
| `default` | `var(--tulpar-button-border-radius)` = 4px | natural | Standard rectangular button |
| `round` | `var(--tulpar-button-radius-round)` = 9999px | natural | Pill |
| `circle` | `9999px` | `1 / 1` | Square + full radius = circle; implies icon-only behavior + warns if no `aria-label` |

`shape="circle"` does NOT set `icon-only` automatically (still allows text inside for design freedom), but the visual reflex (square width = height, no horizontal padding) matches `icon-only`. Use `icon-only` boolean for the strict square-aspect case.

## 8. Loading state

### Behavior

- `loading={true}` suppresses click events (existing behavior).
- `aria-busy="true"` is set (existing).
- The button's content rendering depends on `loading-position`:
  - **`center` (default):** label hidden with `visibility:hidden`, spinner overlaid centered.
  - **`start`:** spinner rendered in the `start` slot position; label remains visible.
  - **`end`:** spinner rendered in the `end` slot position; label remains visible.
- If `loading-label` is set, the visible text becomes `loading-label` instead of the default slot content for any of the three positions.
- The `<slot name="loading-icon">` lets consumers project a custom spinner. If empty, the default Tulpar `.spinner` (existing CSS) is used.

### A11y

- `aria-busy="true"` while loading.
- Default spinner has `aria-hidden="true"`; the loading state is announced via `aria-busy` + the label.
- The `<slot name="loading-icon">` is wrapped in a `<span aria-hidden="true">` so consumer-projected spinner content is excluded from the a11y tree without requiring consumer-side attributes.

## 9. Icons, icon-only mode, and library integration

### 9.1 Architecture — library-agnostic via slots

Tulpar does **not** ship its own icon set. Icons are projected through the `start`, `end`, and `loading-icon` slots from whatever library (or inline SVG) the consumer chooses. The most common candidates:

- **Lucide** — clean line icons, financial-grade, recommended default
- **Heroicons** — outline + solid variants, well-matched to v0.2 typography
- **Tabler Icons** — large library (3000+), MIT licensed
- **Material Symbols** — Google's official set
- **Custom inline SVG** — for brand-specific icons

The Web Component does not know or care which library produced the SVG. It applies sizing normalization via `::slotted()` and centers the icon in its slot area. README + Storybook ship recipes for all four named libraries.

### 9.2 Sizing normalization

Slotted icons are sized via CSS (read `::slotted([slot="start"]), ::slotted([slot="end"]) { width: var(--tulpar-button-icon-size); height: var(--tulpar-button-icon-size); flex-shrink: 0; }`). The `--tulpar-button-icon-size` resolves per the active `size` attribute (xs=12px through 3xl=28px). Consumer overrides via `style="--tulpar-button-icon-size: 20px"` or by setting `width`/`height` directly on their icon element (their explicit value wins over `::slotted()` per CSS specificity rules — note this is a deliberate escape hatch).

### 9.3 `icon-only` mode

- `icon-only` boolean → padding 0, width = height (from the size scale), label hidden visually (default slot content gets `visibility: hidden` if any text leaks in).
- `shape="circle"` → padding 0, width = height, full radius. Implies icon-only visually but does NOT hide the default slot.
- Either flag warns to `console.warn` if neither `aria-label` nor the `label` attribute is present, at `connectedCallback` time. Warning text: `[tulpar-button] icon-only/circle buttons require an aria-label`.
- The warning is dev-time only — no functional block.
- In `icon-only` mode, `icon-position` is ignored (single icon, single position).

### 9.4 `icon-position` — layout direction

```
icon-position: "start" | "end" | "top" | "bottom"   (default: "start")
```

Sets the button's `flex-direction`:

| Value | Direction | Layout |
|---|---|---|
| `start` (default) | row | Icon (start slot) left, label center, icon (end slot) right |
| `end` | row-reverse | Mirrored — start slot right, end slot left |
| `top` | column | Start slot above label, end slot below |
| `bottom` | column-reverse | Start slot below label, end slot above |

For column layouts (`top`/`bottom`):
- The button's height becomes `auto` (the fixed `size` heights only apply to row layouts).
- Padding becomes symmetric (vertical equals horizontal).
- Best with `size="lg"` or larger to maintain a comfortable tap target.
- Use case: toolbar buttons, mobile-style action buttons with stacked label.

If both start AND end slots have content, all four positions still apply (you get icon-label-icon for row, or icon-label-icon stacked vertically for column).

### 9.5 `icon-separator` — visual divider

```
icon-separator: boolean   (default: false)
```

When true, a 1px divider is rendered between the icon area and the label area. Orientation matches the flex direction:
- Row layouts (`start`/`end`) → vertical separator (1px wide, full button height minus padding)
- Column layouts (`top`/`bottom`) → horizontal separator (1px tall, full button width minus padding)

Divider color comes from `--tulpar-button-separator-color`, which resolves per variant: `var(--tulpar-color-border-default)` for outlined/tonal/ghost; `rgba(255,255,255,0.2)` for solid; transparent for link.

When both start AND end slots are used with `icon-separator=true`, two separators are drawn (one per icon). When `icon-only` is set, `icon-separator` has no effect.

## 10. Disabled vs. data-disabled

| State | Behavior | Use case |
|---|---|---|
| `disabled` | Native `disabled` on `<button>`; on `<a>`, sets `aria-disabled="true"` + `tabindex="-1"` + `href="javascript:void(0)"`. Blocks pointer events. | Standard disabled |
| `data-disabled` | Sets `data-disabled` attribute. Component CSS styles off `[disabled], [data-disabled]`. Does NOT block pointer events. | Tooltip-wrapped buttons that need to remain hoverable to show a tooltip explaining why; anchor-polymorphic buttons where native `disabled` is invalid |

Both can be set; styling is identical. Click suppression honors `disabled` only.

## 11. ButtonGroup additions

### `stacked` attribute

- `<tulpar-button-group stacked>` lays children vertically (`flex-direction: column`).
- `data-group-position` semantics still apply but radius collapsing is on top/bottom instead of left/right:
  - first: bottom corners square
  - middle: all corners square
  - last: top corners square
- Margin overlap collapses on the top-left axis instead of left.

### Roving tabindex

- Stacked groups use `ArrowUp` / `ArrowDown` for navigation (horizontal groups still use `ArrowLeft` / `ArrowRight`).
- `Home` / `End` keys work in both orientations.

## 12. Migration v0.2 → v0.3

This is a **breaking** API change. Internal-only since v0.2 is not published.

### Variant migration map

| v0.2 | v0.3 |
|---|---|
| `<tulpar-button variant="primary">` | `<tulpar-button>` (severity defaults to primary, variant defaults to solid) |
| `<tulpar-button variant="secondary">` | `<tulpar-button severity="secondary" variant="outlined">` |
| `<tulpar-button variant="danger">` | `<tulpar-button severity="danger">` |
| `<tulpar-button variant="ghost">` | `<tulpar-button variant="ghost">` |
| `<tulpar-button variant="link">` | `<tulpar-button variant="link">` |

### Token migration

- `color.warning.*` → `color.warn.*` (renamed)
- New: `color.info.*`, `color.help.*`, `color.contrast.*` (three new `VariantColorTokens` slots)
- Type `SemanticTokens.color` gains the new fields

### Storybook stories

All v0.2 stories rewritten with the new API. New stories added for:
- `Severities` — all 8 severities × `solid` variant
- `Variants` — all 5 variants × `primary` severity
- `ColorOverride` — gold + 4 other chromatic colors
- `Loading` — three positions × custom spinner
- `Shape` — round + circle + icon-only
- `Stacked group`
