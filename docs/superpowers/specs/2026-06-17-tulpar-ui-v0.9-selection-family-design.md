# Tulpar UI v0.9 ‚Äî Selection Family Design

**Status:** Draft
**Author:** Kaan Akcan
**Date:** 2026-06-17
**Target release:** `@tulpar-ui/*` v0.9.0 (minor pre-1.0 bump ‚Äî 5 new public elements, no breaking changes to existing surface)

## 1. Background & motivation

After Button (v0.1‚Äìv0.4), the Input family (v0.5), and the Shell family + infra (v0.6+), the next foundational primitives are the **Selection family**: boolean and single/multi-choice controls that, alongside the input family, complete the core form vocabulary of every enterprise app. The long-term roadmap targets (data grid, Gantt, configurator panel) all sit on top of these primitives ‚Äî e.g., the deferred configurator panel is explicitly blocked on "Select/Switch/Checkbox first".

Every decision in this spec is grounded in a per-element cross-industry survey of the same 10 enterprise design systems used for the input family ‚Äî PrimeNG, Ant Design, Carbon, Fluent UI v9, Material UI, Adobe Spectrum, Atlassian Design System, Mantine, Vaadin, SLDS ‚Äî plus **GitHub Primer** where it is the clearest reference (switch loading state). Per-system findings and the synthesis behind specific choices live in the 2026-06-17 brainstorming transcript; this spec records the conclusions only.

## 2. Scope

### In scope (v0.9)

Five new public custom elements, sharing two unexported abstract bases:

1. **`<tulpar-switch>`** ‚Äî boolean on/off toggle. Differentiator: `loading` (async-pending) state; opt-in customizable thumb icons; per-state color override.
2. **`<tulpar-checkbox>`** ‚Äî boolean checkbox with presentational `indeterminate` third state; selectable **Card** variant.
3. **`<tulpar-radio>`** ‚Äî single radio control, meaningful inside a `<tulpar-radio-group>`; selectable **Card** variant.
4. **`<tulpar-radio-group>`** ‚Äî single-select group; roving-tabindex + arrow-key navigation; orchestrates `value`, `name`, `size`, status propagation.
5. **`<tulpar-checkbox-group>`** ‚Äî multi-select group; `value: string[]`; independent Tab/Space per child; consumer-driven "select all".

### Out of scope (deferred)

- **Track-internal on/off text** (Ant `checkedChildren` / `unCheckedChildren` text labels, Carbon small-toggle state text). Localization-fragile (TR "A√áIK/KAPALI" overflows), harms alignment. Thumb **icons** are supported instead (see ¬ß5.3); track text is not.
- **`max-selected`** on CheckboxGroup (Mantine `maxSelectedValues`). Only one of 10 systems ships it; consumer enforces a cap in their change handler. Non-breaking to add later.
- **Built-in "select all" feature** on CheckboxGroup. The `indeterminate` API + group `value` array let a consumer wire a select-all parent in ~5 lines; we do not ship an opinionated select-all slot/parent that assumes a layout.
- **Segmented control / toggle-button-group.** Belongs to the *button* family, not selection. Future scope; not pulled forward.
- **Tri-state user cycling** on checkbox (click cycling unchecked ‚Üí checked ‚Üí indeterminate). `indeterminate` is presentational/programmatic only; a user click resolves it to `checked` (Spectrum behavior).
- **React wrapper.**

### 3.1 Why not reuse `FormFieldBase`

`FormFieldBase` (the input family's base) is **text-field-shaped**: prefix/suffix lanes, float/notch label positions, mask hooks, a message row bound to a single underlying `<input>`. Selection controls have a fundamentally different anatomy: an inline control box/track + a side label + optional description, no prefix/suffix, no float labels. Inheriting the wrong shape would force `if`-branches throughout.

Instead, two **new** unexported abstract bases mirror the *pattern* of `FormFieldBase` (unexported, abstract, DRY-without-public-inheritance per the existing input architecture decision) without reusing the class. The ~30-line message-row + status-icon renderer is **copied** from `FormFieldBase` per the `_internal` ‚â§50-line copy rule (`docs/architecture/internal-layering.md`) rather than promoted to a shared package ‚Äî no real second cross-package consumer exists yet.

### 3.2 `SelectionControlBase` responsibilities

Unexported, abstract, extends `LitElement`, `static formAssociated = true`, `attachInternals()`. Shared by Switch / Checkbox / Radio.

Owns:

- Form-association plumbing (`name` reflection, `setFormValue`, `formResetCallback`, `formDisabledCallback`).
- Inline anatomy: `label` (attribute + slot), `description` (slot), `label-position ‚àà start | end`.
- Sizing: `size ‚àà xs | sm | md | lg | xl` (matches input family for cross-component alignment).
- State: `disabled`, `readonly`.
- Status: `invalid`, `warn`, plus `helper-text` / `error-text` / `warn-text` message row with layout-shift-safe reservation + `no-message-space` opt-out (copied renderer).
- `required` + `aria-required` wiring.
- Custom-icon slot hook (subclass decides slot names).
- Per-instance color override hook (subclass decides which CSS custom properties).
- Card-variant wrapper hook (Checkbox/Radio override; Switch does not).
- Focus ring, keyboard base (Space toggles), ID generation, ARIA wiring (`aria-describedby`, `aria-invalid`).

### 3.3 `SelectionGroupBase` responsibilities

Unexported, abstract, extends `LitElement`. Shared by RadioGroup / CheckboxGroup. **Not** form-associated itself ‚Äî the slotted child controls carry form values; the group orchestrates them.

Owns:

- `orientation ‚àà vertical | horizontal` (rendered as a flex container; `horizontal` wraps).
- Group label/legend (attribute + slot), `description` (slot).
- `required`, `invalid`, `warn`, `helper-text` / `error-text` / `warn-text` group-level message row (copied renderer), `aria-describedby`.
- Propagation to slotted children of: `name`, `size`, `disabled`, `readonly`, `color`. Children may set their own, but the group is the source of truth and re-applies on slotchange.
- `role="radiogroup"` / `role="group"` + `aria-labelledby` referencing the legend.
- Selection-model hook (subclass): single `value: string | null` vs `value: string[]`.
- Keyboard-model hook (subclass): roving-tabindex + arrow navigation (radio) vs independent focus (checkbox).

Subclasses implement value reconciliation: reading child checked states up into the group `value` and writing the group `value` down into child checked states on change / `slotchange` / `formResetCallback`.

### 3.3.1 Group `change` event contract

Because the groups are **not** form-associated (¬ß12), their `change` event ‚Äî not native form bubbling ‚Äî is the primary integration surface for Vue v-model and Angular `output()`. Both groups dispatch a `change` `CustomEvent` whose `detail` is `{ value }`, where `value` is the group's new value (`string | null` for RadioGroup, `string[]` for CheckboxGroup). The event is `composed: true, bubbles: true` so it crosses the shadow boundary into the wrappers. Individual child controls also emit their own native `change`; the group's `change` is the canonical one consumers bind to.

### 3.4 Public element registration

Each public element calls its own `customElements.define('tulpar-<name>', ...)` guarded by `if (!customElements.get('tulpar-<name>'))` per the v0.4 micro-frontend safety rule. No public element exposes a base type.

| Element | Public value/state contract |
|---|---|
| `<tulpar-switch>` | `checked: boolean` |
| `<tulpar-checkbox>` | `checked: boolean`, `indeterminate: boolean` |
| `<tulpar-radio>` | `value: string`, `checked: boolean` |
| `<tulpar-radio-group>` | `value: string \| null` |
| `<tulpar-checkbox-group>` | `value: string[]` |

No `string | number` unions anywhere in the public API.

## 4. Shared anatomy (SelectionControlBase API)

### 4.1 Label, description, position

```ts
@property({ type: String }) label?: string;
@property({ type: String, attribute: 'label-position', reflect: true })
labelPosition: 'start' | 'end' = 'end';
```

- **Label input ‚Äî attribute vs slot:** `label="Notifications"` for a plain string; `<span slot="label">‚Ä¶</span>` for rich content. If both set, the **slot wins** (consistent with input family).
- **`description` slot:** an optional muted secondary line rendered below the label. `<span slot="description">‚Ä¶</span>`. Used heavily by the Card variant (title + description). Available on all three controls.
- **`label-position`:** `end` (default) places the label/description after the control (LTR: to the right); `start` places it before. The whole label+description block is a click target for the control.
- A control with neither `label` nor `slot="label"` renders no label block; the consumer is responsible for an `aria-label` on the host.

### 4.2 Size scale

```ts
@property({ type: String, reflect: true })
size: 'xs' | 'sm' | 'md' | 'lg' | 'xl' = 'md';
```

Matches the input family's 5-tier scale so a single form can set `size` uniformly across inputs and selection controls and have them align. Control-box / track / dot dimensions scale per tier via tokens; the visual growth of the box is capped at the upper tiers (an `xl` checkbox is not literally `xl`-input height ‚Äî it scales the box + label type, not a giant square). `xs` is the dense/grid tier.

### 4.3 Status & messages

```ts
@property({ type: Boolean, reflect: true }) invalid = false;
@property({ type: Boolean, reflect: true }) warn = false;
@property({ type: String, attribute: 'helper-text' }) helperText?: string;
@property({ type: String, attribute: 'error-text' })  errorText?: string;
@property({ type: String, attribute: 'warn-text' })   warnText?: string;
@property({ type: Boolean, attribute: 'no-message-space' }) noMessageSpace = false;
```

Same precedence as the input family: `invalid && errorText` (`role="alert"`) > `warn && warnText` (`role="status"`) > `helperText` (muted). Message row reserved by default; `no-message-space` removes it for inline contexts. `invalid` drives the control border/box to the error color and sets `aria-invalid="true"`.

Note: the input family's async `validating` state is **not** on selection controls in general ‚Äî only Switch has an analogous `loading` state (¬ß5.2), which is process-specific to a toggle action.

### 4.4 Disabled / readonly

Web-platform semantics, matching the input family: `disabled` ‚Üí no interaction, no submission, `aria-disabled`, muted; `readonly` ‚Üí focusable, submitted, not user-changeable.

### 4.5 Color customization

Default colors come from semantic tokens (brand for the selected/on state, neutral for unselected/off). Per-instance overrides are exposed as thin props that set internal CSS custom properties ‚Äî tokens provide the default, the prop only layers on top (does not break the semantic-token discipline; primitive tokens are never referenced in component CSS).

- **Checkbox / Radio:** `color` ‚Äî the checked/selected fill color (default brand). For the **default token-driven** color, the glyph (check/dot) contrast color is paired in the brand layer to meet AA. When a consumer supplies an **arbitrary** `color`, the glyph defaults to a fixed on-color (white/near-white) and contrast against that custom fill is the consumer's responsibility (¬ß11) ‚Äî the component does not attempt a runtime AA computation (which would require a primitive reference, forbidden in component CSS). The Card variant uses `color` for the selected border/accent.
- **Switch:** `on-color` and `off-color` ‚Äî the track color per state (defaults: brand / neutral). The thumb icon color (¬ß5.3) is **derived from the corresponding track color** ‚Äî `on` icon inherits `on-color`, `off` icon inherits `off-color`.
- `disabled` and `invalid` states override these custom colors with the muted/error tokens (accessibility takes precedence over decoration).

CSS custom-property escape hatch is always available for consumers who prefer styling over props (e.g., `--tulpar-switch-track-on`).

## 5. Switch

`<tulpar-switch>` ‚Äî boolean on/off toggle. A switch applies its change immediately (unlike a checkbox, which is staged until form submit).

### 5.1 Public API

```ts
// Inherited from SelectionControlBase: label, description, labelPosition,
// size, disabled, readonly, invalid, warn, required, helperText/errorText/warnText,
// noMessageSpace, name, color override.

@property({ type: Boolean, reflect: true }) checked = false;
@property({ type: Boolean, reflect: true }) loading = false;
@property({ type: Boolean, attribute: 'show-icon', reflect: true }) showIcon = false;
@property({ type: String }) value = 'on';   // form value submitted when checked
@property({ attribute: 'on-color' }) onColor?: string;
@property({ attribute: 'off-color' }) offColor?: string;
// slots: icon-on, icon-off
```

When `checked`, the switch submits `value` (default `"on"`) under its `name`; when unchecked it submits nothing (native checkbox-style FACE behavior).

### 5.2 Loading (async-pending) ‚Äî differentiator

A Tulpar differentiator (only Ant and Primer among surveyed systems ship it). When a toggle triggers a slow async operation:

- `loading=true` shows a spinner in the thumb and sets `aria-busy="true"`.
- The switch **does not visually flip** on its own ‚Äî the consumer sets `checked` only once the operation resolves, then clears `loading`. This prevents the premature on‚Üíoff snap-back that Primer warns against.
- While `loading`, pointer/keyboard toggling is suppressed (the control is busy) ‚Äî but the switch is **not** `disabled`: it stays focusable and `aria-busy="true"` (not `aria-disabled`). `loading` and `disabled` are distinct states.
- The spinner overrides any thumb icon (precedence: `loading` > icon). CSS-based rotation so `prefers-reduced-motion` disables it.

`invalid` / `required` **are** supported (e.g., a required "I accept the terms" toggle) ‚Äî a deliberate divergence from Spectrum, which has no switch error state. The base already provides the plumbing.

### 5.3 Thumb icons (opt-in, customizable)

Default is icon-less (clean track + thumb; state read from color + position + side label).

- `show-icon` (boolean, default `false`) renders an icon inside the thumb. With no slotted icons, the default is a check (`‚úì`) when on and a cross (`‚úï`) when off.
- `slot="icon-on"` / `slot="icon-off"` override the per-state icons (e.g., sun/moon for a theme toggle, lock/unlock). Convenience prop sits on top of slots, never instead of them.
- Icon color derives from the track color (¬ß4.5): on-icon = `on-color`, off-icon = `off-color`.
- Track-internal **text** is explicitly not supported (¬ß2 out of scope).

### 5.4 Anatomy & a11y

- Underlying control uses `role="switch"` with `aria-checked`. Keyboard: Space / Enter toggle; Tab focusable.
- Label/description block is a click target.
- Default sizes ship a visible side label; there is no requirement for state text.

## 6. Checkbox

`<tulpar-checkbox>` ‚Äî boolean checkbox with a presentational third (indeterminate) state.

### 6.1 Public API

```ts
// Inherited from SelectionControlBase (incl. color, description slot).

@property({ type: Boolean, reflect: true }) checked = false;
@property({ type: Boolean, reflect: true }) indeterminate = false;
@property({ type: String }) value = 'on';   // form value when checked (FACE)
@property({ type: String, reflect: true }) variant: 'default' | 'card' = 'default';
// slot: icon  (custom check glyph)
```

### 6.2 Indeterminate

- `indeterminate=true` renders the mixed (dash) glyph and sets `aria-checked="mixed"`, **independent of `checked`** (Spectrum: presentational only).
- It is **programmatic**: a user click resolves the control to `checked=true` and clears `indeterminate`; the consumer re-applies it (this is how a "select all" parent over a CheckboxGroup is wired in ~5 lines).
- 10/10 surveyed systems ship indeterminate ‚Äî mandatory.

### 6.3 Custom icon

`slot="icon"` overrides the default check glyph (Mantine `icon` pattern). The indeterminate dash is not overridable (it carries the `mixed` semantic).

### 6.4 Card variant

`variant="card"` wraps the checkbox + label + description in a clickable bordered card (Mantine `Checkbox.Card`, Spectrum, Ant). The whole card is the click/tap target; `aria-checked` is preserved on the control; checked state styles the card border/background using the `color` value; focus ring is on the card. Inside a horizontal CheckboxGroup, cards lay out as a responsive grid. Default variant remains the plain inline checkbox.

## 7. Radio

`<tulpar-radio>` ‚Äî single radio control. Meaningful only inside `<tulpar-radio-group>`; the group owns selection and keyboard.

### 7.1 Public API

```ts
// Inherited from SelectionControlBase (incl. color, description slot).
// name, size, disabled, readonly, color, invalid are normally inherited
// from the parent group (group is the source of truth) but may be set locally.

@property({ type: String }) value!: string;   // required; identity within the group
@property({ type: Boolean, reflect: true }) checked = false;
@property({ type: String, reflect: true }) variant: 'default' | 'card' = 'default';
```

- `checked` is normally driven by the group from its `value`; setting it directly is reconciled by the group on `slotchange`.
- `variant="card"` mirrors the checkbox Card variant (selected border uses `color`).
- No custom-icon slot ‚Äî the radio dot is the canonical glyph; an icon would read as a checkbox.

### 7.2 A11y

`role="radio"` + `aria-checked`. The group provides roving tabindex (¬ß8.3) so only the checked (or first) radio is tab-reachable; arrow keys move selection within the group.

## 8. RadioGroup

`<tulpar-radio-group>` ‚Äî single-select container.

### 8.1 Public API

```ts
// Inherited from SelectionGroupBase: label (legend), description, orientation,
// required, invalid, warn, helperText/errorText/warnText, noMessageSpace,
// disabled, readonly, size, color, name.

@property({ type: String }) value: string | null = null;
@property({ type: String, attribute: 'name', reflect: true }) name?: string;
@property({ type: String, reflect: true })
orientation: 'vertical' | 'horizontal' = 'vertical';
// default slot: <tulpar-radio> children
```

### 8.2 Selection model

- Single `value: string | null`. Setting it checks the matching child and unchecks the rest.
- A child's user selection sets the group `value` and emits `change` from the group.
- `name` is propagated to all children so native form submission yields a single name/value pair (FACE on the children; the group is not itself form-associated).
- `formResetCallback` on children + group reconciliation restores the initial `value`.

### 8.3 Orientation & keyboard

- `vertical` (default ‚Äî Fluent's scannability/localization rationale); `horizontal` for crowded layouts (wraps).
- Roving tabindex: only the checked radio (or the first, if none checked) has `tabindex="0"`; the rest `-1`. Arrow Up/Left ‚Üí previous, Down/Right ‚Üí next, wrapping; selection follows focus (WAI-ARIA radio group pattern). Home/End jump to first/last.
- `role="radiogroup"` + `aria-labelledby` ‚Üí legend; group-level `aria-describedby` ‚Üí message row.

## 9. CheckboxGroup

`<tulpar-checkbox-group>` ‚Äî multi-select container. The exact parallel of RadioGroup with two differences: array value and independent (non-roving) focus.

### 9.1 Public API

```ts
// Inherited from SelectionGroupBase (same as RadioGroup ¬ß8.1).

@property({ type: Array }) value: string[] = [];
@property({ type: String, attribute: 'name', reflect: true }) name?: string;
@property({ type: String, reflect: true })
orientation: 'vertical' | 'horizontal' = 'vertical';
// default slot: <tulpar-checkbox> children
```

### 9.2 Selection model

- `value: string[]` ‚Äî the `value` of every checked child. Setting it checks the matching children.
- A child toggle adds/removes its `value` from the group array and emits `change` from the group.
- `name` propagation: each checked child submits its own name/value pair (native multi-checkbox form semantics).
- **No roving tabindex** ‚Äî each checkbox is independently Tab-reachable and toggled with Space (WAI-ARIA: checkboxes in a group are not a single tab stop).
- **Select-all** is consumer-driven: a separate `<tulpar-checkbox>` that lives **outside** the group (it is not one of the group's slotted children and never participates in the group's `value` array). The consumer derives its `indeterminate`/`checked` from `value.length` vs child count, and its `change` sets the group `value` to all or none. The group ships no built-in select-all.

### 9.3 Orientation & a11y

`vertical` default / `horizontal` wraps (same as RadioGroup). `role="group"` + `aria-labelledby` ‚Üí legend. Card-variant children lay out as a responsive grid in `horizontal`.

## 10. Tokens / CSS variables

All component styling uses semantic CSS custom properties (no primitive token references inside component CSS ‚Äî enforced by ESLint). New surface (illustrative, finalized in implementation):

```
--tulpar-selection-control-bg
--tulpar-selection-control-bg-checked
--tulpar-selection-control-bg-disabled
--tulpar-selection-control-border
--tulpar-selection-control-border-checked
--tulpar-selection-control-border-invalid
--tulpar-selection-glyph            (check / dot / dash color)
--tulpar-selection-focus-ring

--tulpar-switch-track-off
--tulpar-switch-track-on
--tulpar-switch-thumb
--tulpar-switch-thumb-icon-on
--tulpar-switch-thumb-icon-off

--tulpar-selection-card-bg
--tulpar-selection-card-border
--tulpar-selection-card-bg-selected
--tulpar-selection-card-border-selected

--tulpar-selection-label
--tulpar-selection-description
--tulpar-selection-size-{xs|sm|md|lg|xl}      (box/track/dot dimensions per tier)
```

Per-instance `color` / `on-color` / `off-color` props set the `*-checked` / `*-track-*` custom properties inline on the host. All `var()` calls carry a literal fallback per the existing Tulpar pattern. Dark mode + brand follow the existing 3-layer token mechanism (`.dark`, `[data-brand]`).

## 11. Accessibility

- Switch: `role="switch"` + `aria-checked`; `aria-busy="true"` while `loading`.
- Checkbox: `role="checkbox"` + `aria-checked ‚àà true | false | mixed`.
- Radio: `role="radio"` + `aria-checked`; group `role="radiogroup"`, roving tabindex, arrow-key selection.
- CheckboxGroup: `role="group"`; independent tab stops.
- Groups: `aria-labelledby` ‚Üí legend; `aria-describedby` ‚Üí group message row.
- `aria-required` / `aria-invalid` mirror `required` / `invalid` on controls and groups.
- Error region `role="alert"`, warn region `role="status"`.
- All controls toggle via Space (Switch also Enter); the entire label + description block is a click target.
- **Contrast:** all foreground/background pairs (control border, glyph, focus ring, track, card border, label, description) meet WCAG AA 4.5:1; label/description target AA+ 7:1 across light and dark. Custom `color` / `on-color` / `off-color` values are the consumer's responsibility for contrast, but `disabled`/`invalid` override them with AA-guaranteed tokens. Focus ring verified against the Card variant's filled background and the ghost-like default control background.
- **Reduced motion:** switch thumb slide, loading spinner rotation, checkbox check-draw, card selection transition all respect `@media (prefers-reduced-motion: reduce)` (snap / no continuous rotation).

## 12. Form association

Switch / Checkbox / Radio are form-associated custom elements (`static formAssociated = true`, `attachInternals()`), continuing the v0.4/v0.5 pattern:

- `name` / value reflection; `setFormValue` on every change (checkbox submits its `value` when checked; switch submits `"on"`/configurable when checked; radio submits its `value` when checked).
- `formResetCallback()` ‚Üí reset to initial state.
- `formDisabledCallback()` ‚Üí propagate fieldset disabled.
- Groups are **not** themselves form-associated ‚Äî they orchestrate slotted children, whose FACE participation bubbles to native `<form>` / Angular Reactive Forms / Vue binding without extra plumbing. No `ControlValueAccessor` in the Angular wrapper.

## 13. Wrappers

### 13.1 Angular (`@tulpar-ui/angular`)

Signal-based wrappers per the Button/Input precedent (`input()` / `output()` / `signal()` only ‚Äî **no RxJS in deps**, per project memory):

- `<tulpar-switch-ng>`, `<tulpar-checkbox-ng>`, `<tulpar-radio-ng>`, `<tulpar-radio-group-ng>`, `<tulpar-checkbox-group-ng>`.

Watch the wrapper-nesting gotcha from the shell work (`reference_angular_sidenav_wrapper_nesting`): the `-ng` wrapper wraps the core element, so state attributes (`name`, `size`, `disabled`, `color`) propagated by the group must reach the **core** child, not stop at the wrapper. Group propagation is designed to be wrapper-agnostic (it queries for the core tag and mirrors attributes down); verified in `playground-ng`.

### 13.2 Vue (`@tulpar-ui/vue`)

SFC wrappers with v-model:

- `<TulparSwitch v-model="‚Ä¶" />` (boolean), `<TulparCheckbox v-model="‚Ä¶" />` (boolean), `<TulparRadioGroup v-model="‚Ä¶" />` (string), `<TulparCheckboxGroup v-model="‚Ä¶" />` (string[]).

Consuming apps already configure `compilerOptions.isCustomElement` for `tulpar-` (existing convention).

## 14. Testing strategy

### 14.1 Core (Web Test Runner + open-wc/testing)

Per element:

1. **Rendering & defaults** ‚Äî host, anatomy (label/description slots, attribute-vs-slot, slot wins), default attributes.
2. **Value/state contract** ‚Äî get/set, reflection, FormData submission, FACE compliance, `formResetCallback`.
3. **Status states** ‚Äî invalid/warn, message-row precedence + replacement, ARIA wiring, `no-message-space`.
4. **Keyboard & pointer** ‚Äî Space/Enter toggle, label/description click target, disabled/readonly inertness.
5. **Color override** ‚Äî `color` / `on-color` / `off-color` set custom properties; disabled/invalid override them.
6. **Per-element specifics:**
   - Switch: `loading` (no auto-flip, `aria-busy`, toggle suppression, spinner > icon precedence), `show-icon` default + slotted `icon-on`/`icon-off`, icon color derivation, `invalid`/`required`.
   - Checkbox: `indeterminate` (`aria-checked="mixed"`, independence from `checked`, user-click resolution), custom `icon` slot, Card variant (click target, selected styling, focus).
   - Radio: requires `value`, group-driven `checked`, Card variant.
   - RadioGroup: single `value` reconcile, `name`/`size`/`color`/`disabled` propagation, roving tabindex + arrow/Home/End nav, orientation, slotchange re-reconcile.
   - CheckboxGroup: array `value` reconcile, independent Tab/Space, propagation, consumer select-all wiring (indeterminate parent), orientation.

Shell-test gotcha awareness (`reference_wtr_shell_test_gotchas`): no viewport-dependent assumptions here, but keep zombie-Chrome/port hygiene.

### 14.2 Wrappers (Vitest + happy-dom)

Smoke only: element registers, defaults pass through, two-way binding round-trips (boolean / string / string[]), form-integration smoke (group `change` emission). Visual integration verified manually in `playground-ng` / `playground-vue` (including the group‚Üícore attribute-propagation check in Angular per ¬ß13.1).

### 14.3 Coverage

## 16. Release & sequencing

### 16.1 Branch model

Long-lived `release/v0.9-selection-family`. Topic branches:

- `feat/v0.9-selection-bases` (both internal bases + tokens; no public define)
- `feat/v0.9-switch` (depends on control base)
- `feat/v0.9-checkbox` (depends on control base)
- `feat/v0.9-radio` (depends on control base)
- `feat/v0.9-radio-group` (depends on group base + radio)
- `feat/v0.9-checkbox-group` (depends on group base + checkbox)
- `feat/v0.9-card-variant` (checkbox + radio card styling)
- `feat/v0.9-wrappers` (Angular + Vue)

Each merges into the release branch; release branch ‚Üí `main` as a single minor bump (v0.9.0). Per project memory, no git worktrees ‚Äî normal branches in the main working dir; Kaan merges from GitHub.

### 16.2 Build order (recommended)

1. **Bases + tokens.** Both abstract bases, copied message-row/status renderer, token surface, no public define.
2. **Switch.** Validates `SelectionControlBase` end-to-end + the differentiators (loading, icons, colors).
3. **Checkbox.** Adds indeterminate + custom icon.
4. **CheckboxGroup.** Validates `SelectionGroupBase` with the simpler (independent-focus, array) model first.
5. **Radio + RadioGroup.** Adds roving tabindex + arrow navigation (the most involved keyboard model).
6. **Card variant.** Layered onto Checkbox + Radio once their default variants are solid.
7. **Wrappers + Storybook.**

Visual design pass (track/thumb proportions, card styling, motion curves, exact per-tier sizing) happens during implementation via `/frontend-design` + `/ui-ux-pro-max`, exactly as the input family did.

### 16.3 Changesets

One `minor` changeset listing the 5 new components + the two internal bases.

### 16.4 Migration guide

`docs/migration/v0.x-to-v0.9.md` ‚Äî additive, no breaking changes. Examples: settings form (switches), terms-acceptance (required switch/checkbox), multi-select filters (CheckboxGroup + select-all), single-choice card picker (RadioGroup card variant), async toggle (switch loading).
