# Tulpar UI v0.4 — Enterprise Hardening Design

**Status:** Draft
**Author:** Kaan Akcan
**Date:** 2026-06-08
**Target release:** `@tulpar-ui/*` v0.4.0 (major bump, pre-1.0 breaking changes allowed)

---

## 1. Background & motivation

A v0.3.1 code review surfaced 30 items spanning real bugs, enterprise-grade gaps, and quality-of-life improvements. This spec addresses **items 1–20** — every bug plus every enterprise gap. Items 21–30 (polish, tooling, docs depth) are deferred to v0.5+.

The v0.3.x line shipped the Button surface; v0.4 hardens it so the component can be trusted in production form flows, CSP-strict environments, accessible toolbars, and micro-frontend bundle setups.

## 2. Scope

### In scope (20 items)
- **Bugs (6):** form double-submit (#1), Vue auto-icon-only false-negatives (#2), `javascript:void(0)` href on disabled anchor (#3), ButtonGroup selector grabs nested wrappers (#4), focus delegation broken (#5), ButtonGroup missing mutation observer (#6).
- **Enterprise gaps (14):** double `customElements.define` crashes (#7), hardcoded color fallbacks drift (#8), focus ring contrast (#9), tooltip limitations undocumented (#10), `console.warn` in production bundles (#11), missing native form `name`/`value` parity (#12), type-union triplication (#13), type-only imports across wrappers (#14), `ng-package.json` schema path fragility (#15), Angular wrapper has only smoke tests (#17), Vue test uses invalid variant value (#18), click listener attached in constructor (#19), keyframe global-namespace naming (#20).

### Out of scope (deferred)
- #16 sourcemap/declaration map audit — v0.5.
- #21 CSS-only color override refactor — large lift, deferred.
- #22 lucide-angular optional peer dep — v0.5.
- #23 Storybook setup — separate initiative.
- #24 ESLint plugin expansion — v0.5.
- #25–26 CI/CD workflows + changeset config — separate ops initiative.
- #27–28 wrapper micro-improvements — addressed indirectly by #13.

### Out of scope (explicitly v0.5)
- Tooltip rewrite via Popover API (#10 is documentation-only here).
- Form validation API surface (`setValidity`, custom error messages).
- React wrapper.
- Second brand theme.

## 3. Release & sequencing strategy

- **Branch model:** long-lived `release/v0.4-hardening`. Each group merges via a topic branch (`feat/v0.4-<group>`) → release branch. Release branch → `main` at the end as a single major bump.
- **Changeset:** one `major` changeset covering all 20 items, listed as a migration guide bullet list.
- **Migration guide:** `docs/migration/v0.3-to-v0.4.md` documents every observable behavior change.
- **Versioning:** linked `@tulpar-ui/*` bump to `0.4.0`.
- **Test discipline:** each group's PR adds tests for its items. Existing 80% threshold preserved.

## 4. Groups & sequencing

The 20 items split into 5 thematic groups, sequenced for low blast radius first:

| # | Group | Items | Risk |
|---|---|---|---|
| 1 | Form & native parity | #1, #12 | Medium — observable form behavior changes |
| 2 | Security & lifecycle | #3, #7, #11, #19 | Low — additive guards + namespacing |
| 3 | ButtonGroup correctness | #4, #5, #6, plus wrapper transparency, #29 (host role) | Medium — touches public DOM contract |
| 4 | DX & type dedup | #13, #14, #15, #17, #18 | Low — internal cleanup + tests |
| 5 | A11y, fallbacks, reactive fixes | #2, #8, #9, #10, #20, #30 | Low–medium — visual + a11y changes |

Note: #29 (host `role="group"`) and #30 (loading `aria-live`) were in the review's "improvements" tier but rolled into v0.4 because they sit naturally in groups 3 and 5.

## 5. Group designs

### Group 1 — Form & native parity (#1, #12)

**Goal:** `<tulpar-button>` matches native `<button>` semantics in forms.

**#1 Double-submit fix**
- Shadow-rendered button uses `type="button"` unconditionally. Form actions are driven only from host's `_handleClick`.
- Host listener calls `e.preventDefault()` on submit/reset to absorb any inadvertent native dispatch, then calls `form.requestSubmit(this)` or `form.reset()`.

**#12 Native `name` / `value` parity**
- Add reflective props:
  ```ts
  @property({ type: String, reflect: true }) name?: string;
  @property({ type: String, reflect: true }) value?: string;
  ```
- In `connectedCallback` and on `name`/`value` changes, call `this._internals.setFormValue(this.value ?? null)`.
- `form.requestSubmit(this)` passes the host as `submitter`, so the form serializes `name`/`value` from the host.

**Tests added**
- `formData.get(name) === value` after submit.
- Exactly one submit event per click (use counter).
- `type=reset` resets without submitting.
- Disabled + submit → no submit (preserves existing test).

### Group 2 — Security & lifecycle (#3, #7, #11, #19)

**#3 Drop `javascript:void(0)`**
- Render `<a href=${this.disabled ? undefined : this.href}>`. Lit omits the attribute when `undefined`. CSP-strict environments no longer break. `aria-disabled` + `tabindex="-1"` + `_handleClick`'s `preventDefault` already prevent activation.

**#7 Define-guard for micro-frontend safety**
- Replace `@customElement("tulpar-button")` with:
  ```ts
  if (!customElements.get("tulpar-button")) {
    customElements.define("tulpar-button", TulparButton);
  }
  ```
- Same for `tulpar-button-group`. Removes the decorator import; class still uses `@property`.

**#11 Dev-only warning helper**
- New helper `packages/core/src/_internal/warn-dev.ts`:
  ```ts
  export function warnDev(msg: string, ctx?: unknown): void {
    const isDev =
      (typeof import.meta !== "undefined" && (import.meta as any).env?.DEV) ||
      (typeof process !== "undefined" && process.env?.NODE_ENV !== "production");
    if (isDev) console.warn(msg, ctx);
  }
  ```
- All `console.warn` callsites route through this. Bundlers tree-shake the conditional in production.

**#19 Symmetric click-listener lifecycle**
- Move `addEventListener("click", ...)` from constructor into `connectedCallback`. Mirror with `removeEventListener` in `disconnectedCallback`. Attach with `{ capture: true }` so host intercepts before shadow `<button>` would emit native form events.

### Group 3 — ButtonGroup correctness (#4, #5, #6, wrapper transparency, #29)

**Wrapper transparency (prerequisite)**
- Every Angular wrapper host gets `:host { display: contents }`. The wrapper's element is preserved for component identity but removed from layout/box tree. ButtonGroup's `:scope > tulpar-button` then finds the real button in nested wrapper setups.
- Document in [packages/angular/README.md] under "Layout transparency."

**#4 Direct-child scoping**
- ButtonGroup uses `:scope > tulpar-button` (not `querySelectorAll("tulpar-button")`). Nested groups don't double-claim children.

**#5 Focus delegation**
- ButtonGroup adds `static shadowRootOptions = { mode: "open", delegatesFocus: true }` to `TulparButton`. Roving tabindex applied to host; focus reaches shadow `<button>` via delegation.
- Preserves existing roving-tabindex test semantics.

**#6 MutationObserver-driven sync**
- Replace `queueMicrotask` initialization with `MutationObserver(childList: true)`. Single `_sync()` method runs tabindex init + child positioning. Connected once, disconnected on `disconnectedCallback`.
- Result: dynamic insert/remove/reorder of `<tulpar-button>` children keeps tabindex + group-position attributes consistent.

**#29 Host `role="group"`**
- Constructor: `this._internals = this.attachInternals(); this._internals.role = "group";`. Remove `role="group"` from shadow `<div>` to avoid double role.

**Tests added**
- Insert a button after mount → tabindex re-balances.
- Remove middle button → group-position attributes update.
- Wrapper-transparency: synthetic test with a "fake wrapper" `<div style="display: contents">` containing `<tulpar-button>` confirms group still detects it.

### Group 4 — DX & type dedup (#13, #14, #15, #17, #18)

**#13 Core as single source of truth for types**
- `packages/core/src/button/index.ts` exports the full type set:
  ```ts
  export type {
    ButtonSeverity, ButtonVariant, ButtonSize, ButtonShape,
    ButtonIconPosition, ButtonLoadingPosition, ButtonType,
    ButtonJustify, ButtonColor
  } from "./tulpar-button";
  ```
- Wrappers delete local type unions, import from `@tulpar-ui/core` (peer dep already in place).

**#14 Type-only import enforcement**
- ESLint config (`@typescript-eslint/consistent-type-imports`) already on. After #13 the new imports are type-only and lint passes. No config change needed.

**#15 `ng-package.json` schema path**
- Delete the `$schema` line. ng-packagr accepts the config without it. IDE autocomplete loss is tolerable; the gain is robustness across pnpm hoisting modes.

**#17 Real Angular wrapper tests**
- Adopt `@analogjs/vitest-angular` for TestBed-backed Vitest. Setup file: `packages/angular/vitest.setup.ts`.
- Tests:
  - `effectiveIconSize` computed: default per `size`, override via `iconSize`.
  - `autoIconOnly` computed: true when `icon` set + no projected content; false when content present.
  - Host attribute reflection: `severity`, `variant`, `disabled`, `block` flow to inner `<tulpar-button>`.
  - `clicked` output emits on inner click.
- Existing smoke tests preserved as a safety net.

**#18 Fix invalid variant in Vue test**
- `packages/vue/src/button/TulparButton.test.ts:18` — `variant: "danger"` → `variant: "outlined"`. Update assertion to match.

### Group 5 — A11y, fallbacks, reactive fixes (#2, #8, #9, #10, #20, #30)

**#2 Reactive auto-icon-only**

*Vue:*
```ts
const autoIconOnly = computed(() => {
  if (!props.icon) return false;
  const slot = slots.default?.();
  if (!slot || slot.length === 0) return true;
  return !slot.some(vnode =>
    vnode.type !== Comment &&
    !(typeof vnode.children === "string" && !vnode.children.trim())
  );
});
```

*Angular:* replace `ngAfterContentInit` one-shot read with a MutationObserver on the host element (observe `childList` + `characterData` + subtree). Update the `hasText` signal. Disconnect in `ngOnDestroy`.

**#8 Token-derived fallbacks**
- New build step in `@tulpar-ui/tokens`: `build/generate-fallbacks.ts` reads `primitive/color.ts` and emits `packages/core/src/_fallbacks.generated.ts`:
  ```ts
  export const FALLBACKS = {
    borderDefault: "#e7e5e4",
    textPrimary: "#1c1917",
    textInverse: "#fafaf9",
  } as const;
  ```
- `tulpar-button.styles.ts` uses `unsafeCSS(FALLBACKS.borderDefault)` inside `var(--tulpar-..., ${unsafeCSS(...)})`.
- Core's build script runs the generator before `vite build`. Generated file is git-ignored or committed (chosen at implementation; default: committed, simpler review).

**#9 Variant-aware focus ring**
- New focus-visible style:
  ```css
  .btn:focus-visible {
    outline: none;
    box-shadow:
      0 0 0 2px var(--tulpar-color-bg-surface, ${unsafeCSS(FALLBACKS.bgSurface)}),
      0 0 0 4px var(--_btn-color-active);
  }
  ```
- Outer 2px halo of surface color separates the ring from the button; inner 2px uses the variant's active color, guaranteeing contrast against any background.
- `--tulpar-color-focus-ring` token preserved as an opt-in override for consumers who want a custom system-wide focus color.

**#10 Tooltip limitations docs**
- `tulpar-button.ts` JSDoc on the `tooltip` property:
  ```
  Simple inline string tooltip. Limitations:
  - Clipped by ancestors with `overflow: hidden`.
  - Uses a fixed z-index (100); may be obscured by modals/drawers.
  - No ESC dismiss, no hover delay, no viewport-edge collision detection.
  For production needs, use a dedicated tooltip component (planned v0.5).
  ```
- Mirror in `README.md` under a new "Tooltip" subsection.
- No code change.

**#20 Keyframe namespace**
- `@keyframes tulpar-spin` → `@keyframes tulpar-button-spin`. Update both the keyframe definition and the `animation:` shorthand consumer.

**#30 Loading `aria-live`**
- The loading-label-text span gets `aria-live="polite"`. When `loading-label` is set and `loading` toggles on, SR readers announce the label. `polite` chosen to avoid interrupting typing.

## 6. Migration guide outline

`docs/migration/v0.3-to-v0.4.md` will cover:

- **Form behavior:** clicking a disabled `<tulpar-button type="submit">` no longer dispatches phantom submit events. Apps relying on the previous double-fire need updates.
- **Disabled anchor href:** disabled `href`-buttons no longer set `href="javascript:void(0)"`. Anchor still inert via `aria-disabled` + `tabindex="-1"`.
- **New `name` / `value`:** form submission now carries the button's `name`/`value` like native `<button>`. Forms that listened to `submit` events and got no name/value previously will see them now.
- **Wrapper DOM:** Angular wrapper hosts gain `display: contents`. Visual layout unchanged; CSS that targets `tulpar-button-ng { ... }` as a layout box must be reviewed.
- **Type imports:** wrappers' inline types are removed. Consumers importing from wrapper packages keep working via re-exports; explicit migration: `import type { ButtonSeverity } from "@tulpar-ui/core"`.
- **Console output:** development-mode warnings unchanged; production bundles no longer log button-validation warnings.

## 8. Risks & mitigations

- **Display:contents browsers** — supported everywhere except very old browsers; documented as a v0.4 baseline requirement (Chromium 65+, Firefox 78+, Safari 12+).
- **delegatesFocus** — well-supported; preserves nested focus rings on inner button. No degradation.
- **Form parity** — apps that worked around the missing `name`/`value` by setting hidden inputs will now see duplicate values in formData. Migration guide flags this.
- **Generated fallbacks** — committed `_fallbacks.generated.ts` could go stale if a contributor edits `primitive/color.ts` without re-running the build. Mitigation: CI step (future) or pre-commit hook (deferred to v0.5).
