# Tulpar UI v0.8 — Sidebar Bug Fixes + Rail Flyout

**Date:** 2026-06-17
**Status:** Approved (design phase)
**Branch:** `feat/v0.8-shell-sidebar-redesign`
**Scope owner:** Kaan Akcan

## Summary

Six sidebar issues surfaced from real playground usage (Vue + Angular). Four are
quick, well-understood bugs; two (rail hover discoverability + rail group toggle)
are one underlying problem — **the collapsed icon-rail has no real way to reveal
and navigate a nav group's children.** This spec fixes the four bugs and replaces
the rail's label-only tooltip with an enterprise-standard **flyout submenu**.

All six ship in **one feature branch, one changeset, one PR**.

The rail-flyout design was researched against ~10 leading enterprise products
(GitLab, Atlassian/Jira, Azure Portal, Google Cloud Console, Datadog, Grafana,
ServiceNow, PatternFly, VS Code, Linear) via two independent design passes
(`frontend-design` + `ui-ux-pro-max`). Both converged on the same model: a
JS-positioned flyout panel, hover-intent + click-to-pin, using the **Disclosure**
ARIA pattern (NOT `role="menu"`), with inline expansion suppressed-but-preserved
when entering rail.

## Out of scope

- Multi-level (grandchild) cascading flyouts — keep nav strictly **one level**
  deep (PatternFly's explicit >2-level warning). Future scope if ever needed.
- Azure-style user preference toggle between "hover flyout" vs "docked" rail.
- Full-sidebar "edge peek" overlay (GitLab style) — targeted per-group flyout only.
- Any new public component; this is all inside `@tulpar-ui/shell` + wrappers + playgrounds.

## Part A — Four quick fixes

### A1. Parent (group) item icon↔label gap (#1)

**Symptom:** A nav item that has children (e.g. "Form Inputs") renders with an
oversized gap between its icon and its label text; leaf items align fine.

**Root cause:** In [`tulpar-nav-item.styles.ts`](../../../packages/shell/src/nav-item/tulpar-nav-item.styles.ts)
the `.label` uses `flex: 1` and the trailing `.chevron` uses
`margin-inline-start: auto`. The interaction between these (plus the empty
`<slot name="trailing">`) mis-distributes free space for items that render a
chevron, pushing the label off the icon.

**Fix — commit to this CSS invariant** (gives A1 a verifiable contract, not just
"looks aligned"): *the label is the sole `flex: 1` child; exactly one auto-margin
source positions trailing content; group and leaf items share an identical leading
offset.* Since `.label { flex: 1 }` already consumes free space and pushes trailing
content to the edge on its own, every trailing element's `margin-inline-start: auto`
(`.badge`, `.count`, `.kbd-hint`, `.dot`, `.chevron`) is redundant when a flexed
label is present — the likely gap source is the group's chevron `margin-inline-start:
auto` fighting the flexed label (double auto-margin), and/or the empty
`<slot name="trailing">` as a flex child. Resolve to the invariant above; verify
visually in both playgrounds. Must not regress leaf-item layout, badge/count/dot
trailing elements, or RTL (`margin-inline-*` logical props are retained).

**Acceptance:** In a side-by-side render, a group item's label starts at the same
x-offset as a sibling leaf item's label; the chevron sits at the trailing edge;
badge/count/dot still align trailing on leaf items.

### A2. Theme toggle label always says "Theme" (#2)

**Symptom:** The built-in utility theme button always reads "Theme" regardless of
mode. Requirement: by default show the **target** mode (the mode you switch TO),
but remain overridable.

**Root cause:** Hardcoded `<span class="util-text">Theme</span>` in
[`parts/utility.ts`](../../../packages/shell/src/sidenav/parts/utility.ts); the
sidenav has no awareness of current dark/light state.

**Design — "show target mode":**
- Light mode → button reads **"Dark"** (click → go dark).
- Dark mode → button reads **"Light"** (click → go light).
- Two new sidenav props, both overridable:
  - `theme-text-dark` (default `"Dark"`) — text shown while in **light** mode.
  - `theme-text-light` (default `"Light"`) — text shown while in **dark** mode.
  - Naming is **settled** (keep these, not a `-to-` variant, which reads as an
    action and is ambiguous about which mode it appears in). `theme-text-dark` =
    the word shown that means "switch to dark."
- The sidenav must know the current mode. The shell already owns `dark` and
  toggles `.dark` on `<html>`. Wiring requires **two call sites** (not one):
  - (a) Reflect `data-dark` onto the slotted sidenav from inside the
    `changed.has("dark")` branch of `TulparShell.updated()` — note
    `_updateStateAttrs` is currently NOT called from that branch, so this is a
    new call, parallel to the `data-rail`/`data-collapsed` reflection pattern.
  - (b) Also sync `data-dark` on initial sidenav slot attach (in/near
    `_observeSidenavSlot` / `_updateStateAttrs`) so a page loaded already in dark
    mode shows the correct label before any toggle.
  - The sidenav reads `data-dark` to choose which label + icon (moon/sun) is shown.
- The moon/sun icon swap already exists in `utility.ts` but is currently driven by
  `:host-context(.dark)` (`utility.styles.ts`). **Do not run two parallel
  mechanisms** — drive BOTH icon and text from the single `data-dark` signal and
  drop the `:host-context(.dark)` icon rule, or icon/text can desync on the exact
  transition frame (the very bug A2 is meant to kill).

**Acceptance:** With no overrides, toggling the built-in button flips the label
between "Dark" (in light) and "Light" (in dark), with the icon matching. Setting
`theme-text-dark="Karanlık"` overrides the light-mode label. Works in both
playgrounds and Storybook.

### A3. Angular playground body padding (#5)

**Symptom:** The Angular playground page layout breaks "before even reaching the
shell" — content is inset and the shell can't fill the viewport.

**Root cause:** [`apps/playground-ng/src/styles.css`](../../../apps/playground-ng/src/styles.css)
sets `body { padding: 2rem }` — leftover from the pre-shell demo. The Vue
playground correctly uses `body { margin: 0 }` with no padding.

**Fix:** Remove the body padding (and confirm `margin: 0`) so the shell fills the
viewport, matching Vue.

**Acceptance:** Angular playground shell fills the viewport edge-to-edge with no
outer gap; visually matches the Vue playground framing.

### A4. Angular groups render expanded, no chevron, won't collapse (#6)

**Symptom:** In the Angular playground, a nav item with children ("Form Inputs")
renders already-expanded, shows no chevron, and offers no way to collapse.
Vue works correctly.

**Root cause:** Core `TulparNavItem._onSlotChange`
([`tulpar-nav-item.ts`](../../../packages/shell/src/nav-item/tulpar-nav-item.ts))
detects children by matching `el.tagName === "tulpar-nav-item"`. The Angular
wrapper nests children as `<tulpar-nav-item-ng>` (a `display:contents` host that
wraps the real `<tulpar-nav-item>`), so the assigned element's tag is
`tulpar-nav-item-ng` → `_hasChildren` stays `false` → no chevron, and
`?hidden=${this._hasChildren && !this._expanded}` is always `false` so children
never hide. Vue nests the bare element, so its detection passes.

**Fix:** Make child detection wrapper-agnostic. An assigned element counts as a
child when it **is** a nav item or **contains** one:
```
el.matches("tulpar-nav-item, tulpar-nav-item-ng") || el.querySelector("tulpar-nav-item")
```
This keeps core authoritative while tolerating the Angular wrapper's extra
`display:contents` layer, without core importing wrapper names as a hard
dependency (the strings are matched, not imported). The same matcher is reused
anywhere the sidenav enumerates nav items if a parallel gap exists
(`_focusableItems`, `_reflectRail` already query `tulpar-nav-item` in the
shadow/light DOM — verify they still resolve the bare elements, which they do
since the real element is present; only the *child-of-parent* slot detection is
broken).

**Acceptance:** Angular playground "Form Inputs" renders collapsed by default with
a chevron; clicking toggles it; `singleExpand` and keyboard expand/collapse work
identically to Vue. Existing Vue/core behavior unchanged.

## Part B — Rail flyout (enterprise — #3 + #4)

### B0. Problem

In rail (collapsed icon-only) mode:
- **#3:** Hovering an icon shows only a tiny label tooltip. For a **group**, its
  children are completely unreachable — no way to navigate into them.
- **#4:** A group expanded before collapsing to rail stays "expanded" (children
  behave oddly); a group collapsed before railing can't be opened. There is no
  rail-aware toggle.

These are one problem: **the rail needs a real flyout submenu for groups, and
inline expansion must be rail-aware.**

### B1. Interaction model

**Leaf item (no children):** unchanged — the existing `.rail-flyout` label tooltip.

**Group item (has children):** rail hover/focus/click opens a **flyout panel**:
- Header row: the group's label (+ its icon), non-interactive (or a link if the
  group itself is navigable).
- Body: the group's child items as real `<a>` links; the active child carries
  `aria-current="page"` and the active treatment (accent bar) used inline.
- The flyout is a strict **superset** of the leaf tooltip — a group shows the
  flyout, never both flyout and tooltip. **Implementation:** branch a single
  rail-overlay template by `_hasChildren` (label-only tooltip vs flyout panel)
  rather than rendering two separate elements; this keeps the single-flyout
  coordination state in `tulpar-sidenav.ts` and the `role`/`aria` wiring (tooltip
  vs `role="group"`) on one node.

**Triggers (co-equal):**
- **Hover (mouse):** `pointerenter` arms an open timer; flyout opens after a
  **120 ms** intent delay. `pointerleave` arms a **240 ms** close grace.
- **Focus (keyboard/AT):** opens **immediately (0 ms)** — intent already committed.
- **Click / tap:** **pins** the flyout open (does not navigate — a group trigger
  is a disclosure, not a link).

**Safe triangle:** while the close timer runs, compute the triangle between the
current cursor and the flyout's two near corners; if the next pointer move stays
inside it, cancel the close. Solves the diagonal-travel dismiss problem
(Amazon mega-dropdown technique).

**Dismiss:** `Esc`, outside click, opening another icon's flyout, or
scroll/resize closes the flyout. **Only one flyout open at a time.**

**Touch / coarse pointer (`@media (pointer: coarse)` / `matchMedia`):** hover-open
disabled entirely; rail icon is **tap-to-open-and-pin**. Flyout rows ≥ 44px tall;
rail icon hit area ≥ 44×44px.

### B2. Rail state behavior (fixes #4)

- Entering rail mode **suppresses inline expansion** (`:host([data-rail])` →
  inline children subtree `display:none`); children render **only** in the flyout.
- Each group's expand/collapse state is **preserved, not mutated**, on entering
  rail. Leaving rail restores the prior inline expanded state losslessly.
- Because inline expansion is suppressed (not toggled) in rail, the "stuck
  expanded" and "can't open" states disappear by construction.
- **Focus-scope separation:** `_focusableItems()` filters on `offsetParent !== null`,
  so suppressed (`display:none`) inline children correctly drop out of rail
  arrow-key nav. The flyout's own child links are a **separate** roving-focus
  scope (B4 keyboard model) — `_focusableItems()` is NOT reused for in-flyout
  navigation, so the two scopes never collide.

### B3. Discoverability

- **Persistent chevron cue:** group icons show a small **chevron** (the inline `›`
  glyph, reused for continuity — NOT a dot, which would collide with the existing
  `.dot` status semantics) at the **bottom-right** of the 40×40 icon box with ~2px
  inset, in `--tulpar-shell-sidenav-rail-cue` (`currentColor` at muted opacity).
  Rest opacity ~0.5, hover ~1.0 — discoverable but not noisy. Visible at rest,
  before hover, so the user knows the icon has children.
- **Active-trail:** if the current route is inside a group, that group's rail
  **icon** carries the `--tulpar-shell-sidenav-item-indicator` accent (icon-color
  tint, not a left bar — a bar reads poorly on a 40px square), and the open flyout
  marks the active child with the inline active treatment. The active-child accent
  in the flyout is **static** (no `tulpar-nav-ignite` animation — that's reserved
  for genuine navigation, not flyout open).

### B4. Accessibility (Disclosure pattern, NOT `role="menu"`)

Navigation links are not application menu commands; `role="menu"` mis-announces
links and removes them from Tab order (W3C APG + Adrian Roselli). Use native
semantics with disclosure attributes:

- **Group trigger:** `<button>` with `aria-haspopup="true"`, `aria-expanded`
  (reflects open state), `aria-controls="<flyout-id>"`, and an accessible name
  (`aria-label` = group label, since visible text is hidden at rail width).
- **Flyout panel:** `<div id="<flyout-id>" role="group" aria-label="<group>">`
  containing a `<ul>` of `<a>` links. No `role="menu"`/`menuitem`.
- **Leaf tooltip:** keep `role="tooltip"` + `aria-describedby` wiring.

**Keyboard model:**
| Key | On rail group trigger | Inside open flyout |
|---|---|---|
| Right (LEFT sidenav) / Left (RIGHT sidenav) | open flyout, focus first child | — |
| Enter / Space | toggle/pin flyout, focus first child | activate focused link |
| Up / Down | move between rail icons (existing) | move between child links |
| opposite Arrow / Esc | — | close flyout, **return focus to trigger** |
| Home / End | — | first / last child link |
| Tab | move to next rail item; does **not** auto-open; hidden children not in Tab order | cycles child links naturally |

**WCAG 1.4.13 (content on hover/focus):**
- *Dismissible:* `Esc` closes without moving pointer, restores focus to trigger.
- *Hoverable:* pointer can move icon → flyout and it stays open (safe triangle +
  240 ms grace bridge the 8px gap).
- *Persistent:* stays until pointer leaves both, `Esc`, outside-click, or
  scroll/resize — never auto-times-out while hovered/focused.

### B5. Positioning (reuses existing infrastructure)

The current `position: fixed` + `getBoundingClientRect()` + LEFT/RIGHT flip +
close-on-scroll/resize foundation is correct and reused. Additions:
- **Side:** LEFT sidenav → open right (`left = rect.right + gap`); RIGHT sidenav →
  open left (`right = innerWidth - rect.left + gap`). `gap = 8px`.
- **Vertical:** top-align to trigger; if it would overflow the viewport bottom,
  shift up to fit, clamped 8px from edges.
- **Panel craft:** `min-width` ~200px, `max-width` ~280px (long labels ellipsize,
  not wrap); `border-radius` `0.5rem` (matches `item-radius`, reads more
  substantial than the `0.375rem` leaf tooltip); internal padding 6px block / 6px
  inline, 2px row gap (matches the sidenav's `gap: 2px` rhythm); child rows reuse
  the `item-height`/`item-radius` tokens for visual identity with inline items.
- **Header row:** group label visually distinct from links — slightly smaller,
  `--tulpar-shell-sidenav-fg-muted`, medium weight, with a hairline divider
  (`flyout-border` at reduced opacity) below it, so it doesn't read as a clickable
  row.
- **Elevation — light = shadow-led, dark = border-led** (state this principle
  prominently; it prevents the most common dark-mode failure — a charcoal panel on
  a charcoal sidenav with an invisible shadow). Light: two-layer shadow via
  `--tulpar-shell-sidenav-flyout-shadow` (tight contact + soft ambient, navy-tinted
  per the existing `primitiveShadow.md` language). Dark: collapse the ambient layer
  toward zero and lean on a **brighter** `--tulpar-shell-sidenav-flyout-border`.
  Surface is `--tulpar-shell-sidenav-flyout-bg` — one step **above** the sidenav
  bg (NOT reused `sidenav-bg`, which reads flat in dark).
- **Caret notch:** ~8px-leg triangle on the panel's rail-facing edge, built as a
  rotated square (so both `flyout-bg` fill AND 1px `flyout-border` render — a
  `border-triangle` hack can't carry the border), overlapping the panel edge by 1px
  to hide the seam. Caret Y is decoupled from panel Y: when the panel shifts up to
  avoid overflow, the caret stays pinned to the trigger icon's center, and is
  **clamped within the panel's rounded-corner inset** so it never renders over the
  `border-radius` curve. **RTL/`position="right"`:** the caret edge + direction key
  off `:host([position="right"])` (or a reflected `data-side`) via logical
  positioning — this is the one piece that does NOT "just work" from the JS flip.
- **Hover-bridge:** an invisible hit area spanning the `gap`, with `pointer-events`
  only within the gap band (must not overlap the icon and steal its hover), so the
  safe-triangle rarely needs to fire.

### B6. Motion (premium, reduced-motion aware)

- **Enter:** fade + ~6px slide from the rail edge, ~140 ms, `--tulpar-easing-decelerate`.
  Slide direction is logical: LEFT sidenav from `translateX(-6px)`, RIGHT sidenav
  from `translateX(+6px)` — flips with `position="right"`.
- **Exit:** ~100 ms, faster than enter.
- Animate `transform`/`opacity` only (no layout shift / CLS).
- `@media (prefers-reduced-motion: reduce)` → opacity-only, ≤ 80 ms or instant.
- The originating rail icon stays in hover/active state while its flyout is open.
  **Pinned affordance** (click-pin): trigger keeps the hover bg + the caret remains
  + a subtle 1px ring, so pinned reads as distinct from transient hover.

### B7. Token additions (BLOCKING — required for token-compliant styling)

Component CSS is ESLint-restricted to semantic tokens only (no `--tulpar-primitive-*`).
Today the flyout's needed visual tokens are **referenced but never emitted** — they
silently resolve to inline fallbacks: `--tulpar-shadow-sm`/`-md` and
`--tulpar-easing-decelerate` do not exist in the emitted semantic set
(`packages/tokens/src/semantic/`), and there is no flyout/popover surface distinct
from the sidenav. The primitive *values* exist (`primitiveShadow.sm/md`,
`primitiveTransition.easing.decelerate`) but are unbound. Define and bind these:

| Token | Purpose | Light | Dark |
|---|---|---|---|
| `--tulpar-shell-sidenav-flyout-bg` | flyout surface, one step above sidenav | white/overlay | one step lighter than sidenav bg (not equal) |
| `--tulpar-shell-sidenav-flyout-border` | flyout border; carries elevation in dark | = sidenav border | **brighter/more opaque** than light |
| `--tulpar-shell-sidenav-flyout-shadow` | two-layer elevation | `primitiveShadow.md` (navy-tint) | near-zero (border-led) |
| `--tulpar-shell-sidenav-rail-cue` | group chevron cue color (B3) | muted fg | muted fg |
| `--tulpar-easing-decelerate` | emit the already-referenced easing | `cubic-bezier(0,0,0.2,1)` | same |
| `--tulpar-shadow-md` (or `--tulpar-elevation-popover`) | emit a real shared popover elevation | `primitiveShadow.md` | reduced |

Add the new fields to `ShellTokens.sidenav` in `packages/tokens/src/semantic/shell.ts`
+ its `ShellNavItemTokens`/types, bind them in the brand layer
(`brand/tulpar/light.ts` + `dark.ts`), and extend the tokens tests (`shell.test.ts`).
Emit `--tulpar-easing-decelerate` + `--tulpar-shadow-md` alongside the existing
transition/shadow emission so the existing dangling references in
`tulpar-nav-item.styles.ts` / `tulpar-button.styles.ts` also resolve (fixes a latent
bug, no behavior change since fallbacks matched). Keep additions minimal — only what
the flyout needs; do NOT introduce a full elevation scale (YAGNI).

## Testing strategy

- **Core (Web Test Runner, real Shadow DOM):**
  - A1: group vs leaf label offset.
  - A4: a `<tulpar-nav-item>` with a wrapping child (simulated `display:contents`
    wrapper containing a nav-item) is detected as having children.
  - B1: flyout opens on focus immediately, on hover after intent delay
    (fake timers), pin on click, single-flyout coordination, dismiss on Esc/outside.
  - B2: entering rail suppresses inline children but preserves expanded state;
    leaving rail restores it.
  - B4: ARIA attributes (`aria-haspopup`/`expanded`/`controls`), keyboard
    Right/Left/Enter/Esc focus flow.
  - B1 coarse pointer: with `matchMedia("(pointer: coarse)")` mocked to match,
    hover does NOT open the flyout (tap-to-pin only) — hardens the most
    easily-regressed branch.
  - **WTR gotchas (per MEMORY):** narrow headless viewport flips
    `tulpar-shell._isMobile` — pin `mobile-breakpoint` in viewport-dependent
    rail tests; kill zombie chrome on port 8002 if it wedges; avoid `it.skip`
    bisects that hang.
- **Tokens (Vitest):** `shell.test.ts` asserts the new flyout/cue tokens emit in
  light + dark, and that `--tulpar-easing-decelerate` / `--tulpar-shadow-md` are
  present in the generated CSS (B7).
- **Wrappers (Vitest, smoke):** Angular group toggles; theme-text inputs exported
  and forwarded; Vue theme-text props forwarded.
- **Visual:** both playgrounds gain a rail-group demo; verify A1/A2/A3/A4 and the
  full rail flyout (hover, pin, keyboard, touch via devtools, dark mode, RTL-safe
  logical props, reduced-motion).
