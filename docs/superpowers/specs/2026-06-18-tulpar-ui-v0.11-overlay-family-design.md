# Tulpar UI v0.11 — Overlay Family (Tooltip · Toggletip · Popover)

**Status:** Draft — pending spec review
**Date:** 2026-06-18
**Predecessor:** v0.10 Selection Family (switch/checkbox/radio)
**Type:** New component family + shared internal positioning primitive + tokens + Button integration

---

## 1. Overview

A floating-overlay component family built on a single shared positioning foundation. Three
public custom elements, split by **trigger × interactivity × accessibility role** — the same
axis the leading systems (Carbon, MUI, Radix, React Aria) converge on:

| Element | Trigger | Content | Interactive | Touch | ARIA |
|---|---|---|---|---|---|
| `<tulpar-tooltip>` | hover / focus | brief text | No | degrades to toggletip | `role="tooltip"` + `aria-describedby` |
| `<tulpar-toggletip>` | click (button) | brief text / markup | lightly | Yes | disclosure + `aria-live` |
| `<tulpar-popover>` | click | rich / interactive | Yes | Yes | Dialog (non-modal) + `aria-haspopup` |

A package-private positioning primitive underlies all three (mirrors the input family's internal
`FormFieldBase`). It is **not** a public element and stays narrow: anchor resolution, placement
compute, flip/shift, top-layer management, light-dismiss. ARIA wiring, delay grouping, focus
return, and animation live in the public components — the primitive must not become a god object.

This family also realizes work explicitly deferred in v0.3.1, which shipped a CSS-only string
`tooltip` attribute on Button and promised "rich popovers + collision-aware tooltip = future
`<tulpar-popover>`".

### Goals
- One correct, accessible, on-brand floating layer powering tooltip / toggletip / popover.
- A positioning core that works **today** across the enterprise browser matrix, with native
  enhancements where available.
- WCAG 2.2 AA, with WCAG 1.4.13 (Content on Hover or Focus) satisfied by construction.
- A meaningful, contrast-guaranteed color register that surfaces the brand palette as *intent*.

### Non-Goals (binding)
See §12 Out of Scope.

## 2. Architecture

```
@tulpar-ui/core
  src/_internal/overlay/      ← package-private positioning primitive (positioner)
  src/tooltip/tulpar-tooltip.ts
  src/toggletip/tulpar-toggletip.ts
  src/popover/tulpar-popover.ts
@tulpar-ui/tokens
  src/semantic/overlay.ts      ← new semantic overlay tokens
@tulpar-ui/angular  → tulpar-tooltip-ng / tulpar-toggletip-ng / tulpar-popover-ng
@tulpar-ui/vue      → TulparTooltip / TulparToggletip / TulparPopover
```

The positioner is consumed only within `@tulpar-ui/core` (the `_internal` boundary rule applies —
never re-exported, never imported across packages).

## 3. Positioning engine

A **layered, JS-primary** strategy. The hand-rolled JS engine is the reference implementation and
the guaranteed path (it runs everywhere and is tested first in CI). Native platform features are
progressive enhancements layered on top where supported.

1. **JS flip/shift core (primary):** pipeline `offset → flip → shift → arrow`. Handles the cases a
   "small" helper gets wrong: nested scroll containers, transformed ancestors, subpixel rounding,
   logical-direction (RTL-ready) placement, scroll/resize reflow.
2. **Native Popover API (enhancement):** `popover` attribute → browser top-layer, light-dismiss,
   Esc handling, z-index-war-free stacking. Used when supported; the JS core provides the same
   behavior otherwise.
3. **CSS Anchor Positioning (enhancement only):** enabled behind `CSS.supports('anchor-name')`.
   Never the primary path — it is Chromium-led and only reaching cross-browser baseline in 2026.
   The JS engine remains source of truth; CSS-anchor must produce the same placement decision.

> Rationale: for a paid product, "positioning silently drifts in Safari" is a trust/support-cost
> killer. JS-primary removes that class of bug. We hand-roll rather than depend on `@floating-ui/dom`
> to keep the "from scratch" core promise — but the JS engine is held to Floating-UI-grade
> correctness on the edge cases above (this is a non-negotiable acceptance bar, see §11).

### Placement
- 12 placements: `top | bottom | left | right` × `start | center | end`.
- Internally logical-direction-based (`inline-start`/`inline-end`) so RTL mirroring is a config
  flip later, not a re-architecture — even though full RTL testing stays out of scope (§12).
- `placement="auto"` — best-fit by available space.
- Resolved placement (after flip/shift) reflected as `data-placement` so arrow logic, animation
  origin, and consumer styling stay in sync.

### Positioning props (shared)
`placement, offset, crossOffset, containerPadding, flip, arrow, arrowBoundaryOffset,
boundary` (viewport | scroll-parent | element). Bare-boolean naming (`flip`, `arrow`) to match
Button/Input conventions — not `shouldFlip`/`showArrow`.

### Scroll & resize behavior
- Tooltips: **reposition** while the anchor is visible; **dismiss** when the anchor scrolls out of
  view (a tooltip pointing at nothing is worse than none).
- Popover/toggletip: **reposition** on scroll/resize; **re-measure** when content size changes
  (async-loaded content must not overflow the viewport).

## 4. Composition & API

### Default: wrapper + slots
```html
<tulpar-tooltip>
  <button slot="trigger">Save</button>
  <span slot="content">Saves the current draft</span>
</tulpar-tooltip>
```
- Wrapper uses `display:contents` so it does not disturb layout. **Verify** the slotted trigger
  stays in the accessibility tree and tab order (historical `display:contents` a11y-tree drops).
- The slotted trigger must remain the real focusable element — the wrapper never wraps it in its
  own focusable shell. Dev-mode warning if the trigger slot contains no focusable element
  (WCAG 2.1.1).
- ARIA wiring (`aria-describedby` / `aria-haspopup` / live region) is generated automatically from
  the slots — consumers don't hand-manage ids.

### Optional: anchor-by-id (escape hatch)
```html
<button id="acct">Account</button>
<tulpar-popover anchor="acct"> … </tulpar-popover>
```
Native `popovertarget` philosophy. Secondary mode (the convenience layer sits on top of slots, not
instead of them).

### Open state
- Controlled: `open` (attribute/prop) + `defaultOpen`.
- Events: `tulpar-open` / `tulpar-close` / `tulpar-toggle`.
- Wrappers: Angular signal inputs + `openChange` output; Vue `v-model:open`.

### Imperative API (on the element)
`show()` · `hide()` · `toggle()` · `reposition()` — `reposition()` matters for async content resize.

### Convenience string (tooltip/toggletip)
A `content`/`text` string prop renders simple text without the slot (prop-on-top-of-slot; both
coexist, slot wins when present).

## 5. Accessibility contract (per component)

### Tooltip — `role="tooltip"` + `aria-describedby`
- Shows on hover **and** keyboard focus (`:focus-visible` — not on mouse-click-derived focus).
- Focus stays on the trigger; the tooltip never receives focus and never holds interactive content
  (dev-warn if interactive content is slotted — that's a popover).
- Esc dismisses without moving focus; blur dismisses.
- **WCAG 1.4.13:** dismissible (Esc), hoverable (close-delay + hover-bridge, §6), persistent (no
  auto-hide timeout).
- Only **one** tooltip visible at a time (opening a new one closes the prior).

### Toggletip — disclosure + live region
- Trigger is a real focusable button; Enter/Space toggles; Esc closes and returns focus to trigger.
- Content announced via an `aria-live="polite"` region that **exists in the DOM before** text is
  injected (else SRs miss the first announcement). Focus does not move into the bubble.

### Popover (non-modal) — Dialog pattern
- `aria-haspopup="dialog"`, labelled via slotted heading / `aria-label`.
- On open, focus moves to the popover (first focusable or the container). Tab flows naturally
  (no trap). Light-dismiss on outside click. Esc closes and **returns focus to the trigger**.
- No scroll-lock, no inert background (those belonged to modal — dropped, §12).

### Cross-cutting
- **Every** dismiss path (Esc, outside-click, programmatic) returns focus to the trigger.
- **Disabled triggers:** a real `disabled` button fires no events, so a tooltip on it is invisible
  to everyone — yet "why is this disabled?" is the top tooltip use case. Guidance + components
  prefer `aria-disabled="true"` (focusable, activation suppressed) over `disabled`. Dev-warn when a
  tooltip/toggletip is attached to a `[disabled]` element.
- **forced-colors / HCM:** overlays must not vanish — a real `border` (not shadow alone) defines the
  surface; the arrow stays visible; `tone` (§7) intentionally drops out (author colors overridden).

## 6. Timing & motion

- **Open delay** ~400ms (default; token), short enough to feel responsive, long enough to suppress
  accidental fly-overs.
- **Close delay** ~120ms (token) **+ hover-bridge**: an invisible region spanning the trigger→tooltip
  gap so the pointer can travel onto the tooltip without it closing. Without this the tooltip fails
  1.4.13 "hoverable" (the arrow gap makes it worse). The bridge and arrow are coupled.
- **Delay-group / skip-delay:** moving between sibling tooltips skips the open delay (shared timer
  controller). Prop-controlled, default on. Built in v0.11 (retrofitting it later is a cross-cutting
  refactor).
- **Origin-aware animation:** overlay scales out from the trigger; `transform-origin` follows the
  resolved `data-placement`. Enter ≤200ms, **exit shorter than enter**, interruptible (Esc removes
  it immediately mid-animation; a closing overlay sets `pointer-events:none`).
- **`prefers-reduced-motion`:** disables transform — crossfade/instant, not merely shortened.
- All delays/durations/easings are **motion tokens**, never magic numbers in component CSS.

## 7. Color register — `tone`

A constrained, **semantic** color prop. It surfaces the chromatic-mythology palette as *meaning*,
not raw family names — color must communicate, never just decorate.

```html
<tulpar-popover tone="danger"> … </tulpar-popover>
```

- **Prop:** `tone="neutral | info | success | warning | danger"` (default `neutral`).
- **Applies to:** `<tulpar-toggletip>` and `<tulpar-popover>` only. **Not** `<tulpar-tooltip>` —
  the inverted neutral chip is its identity and its strongest contrast guarantee.
- **Mapping (internal):** each intent maps to one brand family — `info → gök`, `success → ülgen`,
  `warning → kizagan`, `danger → erlik`, `neutral → default surface`. Consumers never type a family
  name; the primitive layer does not leak into the public API.
- **Surface model — soft tonal, auto light/dark flip:**
  | Role | Light | Dark |
  |---|---|---|
  | surface bg | `50`/`100` | `900`/`950` |
  | text (on-surface) | `800`/`900` | `100`/`200` |
  | border | `200` | `800` |
  | arrow | inherits surface + border | — |
  Same-family extreme-vs-extreme pairing targets ~7:1 (AAA) with headroom. **Never** mid-ramp
  (`500`/`600`) fills — that is the AA-failure zone across hues.
- **Focus ring** inside a toned surface stays on a **mode-fixed neutral/brand** token (never the
  tone family) and is verified ≥3:1 against the tinted bg.
- **Guardrails:** a toned overlay **should** carry the matching status icon (WCAG 1.4.1 — color is
  never the sole carrier; documented). `tone` is visual-only — it must not change `role`,
  `aria-live`, or focus semantics. Under forced-colors it is a no-op.
- **Contrast CI test:** all `5 tones × 2 modes × {bg, text, border, focus-ring}` pairs asserted
  ≥4.5:1 body / ≥3:1 large+border+ring before the value is permitted. A failing pair is not shipped.

> Deferred: exposing **arbitrary** brand families as `tone` values, and the tokens-package
> "adaptive mirror layer" that would auto-generate the flip. v0.11 hand-authors the finite tonal
> token set; migrating it onto the adaptive layer later is a non-breaking internal swap (the public
> `tone` enum is unchanged). Widening the enum later is additive/safe; we deliberately start narrow.

## 8. Tokens (new)

Bound in the brand layer (`brand/tulpar/{light,dark}.ts`); component CSS consumes semantic tokens
only (ESLint-enforced; provide `var()` fallbacks).

- `--tulpar-color-overlay-surface` / `-border` — neutral overlay surface (popover/toggletip).
- **Inverted tooltip surface** — dark chip on light theme / light chip on dark (its own token, not
  `bg.surface`); text `text-inverse`.
- A dedicated **overlay shadow ramp** (overlays need elevation distinct from cards).
- `--tulpar-z-overlay` — base z-index tier for the JS-fallback (non-top-layer) path; must sit above
  shell/sidenav.
- `tone` set: `--tulpar-color-overlay-{tone}-surface | -on-surface | -border` × light/dark.
- Motion: `--tulpar-overlay-delay-open | -delay-close | -duration-enter | -duration-exit | -easing`.
- Sizing: tooltip `max-width` (~`32ch`) + wrap; popover `max-height` + internal scroll while
  anchored; tooltip padding tighter than popover padding (4/8px scale).

## 9. Layering & portal boundary

- Native Popover API top-layer handles modern browsers.
- The **JS-fallback path** renders in normal DOM flow and would be clipped by `overflow:hidden` /
  `transform` ancestors — so it portals to a defined root and uses `--tulpar-z-overlay`.
- **Theme inheritance across the portal:** the overlay must inherit `.dark` / `data-brand` context
  regardless of where it is portaled (CSS custom properties cross shadow boundaries, but a
  light-DOM portal must re-establish the brand/mode context). Spec'd as an explicit contract and
  tested in both modes.
- **Nested / stacked overlays:** tooltip-inside-popover is allowed; popover-inside-tooltip is not
  (tooltips hold no interactive content). Esc closes the **top-most** overlay only; light-dismiss
  must not collapse the whole stack. The fallback path has an explicit stack manager.

## 10. Button integration

The v0.3.1 CSS-only `tooltip` string attribute is **refactored to delegate** to `<tulpar-tooltip>`
internally — a single implementation, and a latent bug fix (the CSS-only version isn't hoverable/
dismissible and likely failed 1.4.13). The `tooltip` attribute is **kept** as a convenience prop
(prop-on-top-of-slot) so existing consumers don't break; wiring is lazy/conditional so Button users
who never set `tooltip` pay no overlay cost. Any true removal of the attribute is gated behind a
major changeset + migration note.

## 12. Out of Scope (binding)

- **Modal popover** — dropped. Focus-trap + scroll-lock + inert background belong to a future real
  `<tulpar-dialog>`, not "popover". Popover is non-modal only.
- **Hover-card** (hover-triggered *interactive* card, e.g. GitHub user hover) — distinct pattern,
  future scope; does not fall out of "popover".
- **Dropdown menu / context menu / Select** — future family (needs roving-tabindex menu semantics).
- **Arbitrary brand-family `tone` values** + the tokens adaptive-mirror layer — deferred (§7).
- **Full RTL testing** — placement vocabulary is logical-direction-ready, but RTL verification is
  later (project-wide RTL is out of scope).
- **SSR measure optimization**, **multiple-trigger → one-content**, **virtualized long popover
  lists**, **truncated-text-only tooltip convenience** — deferred; the primitive API must not
  foreclose them.

## 13. Docs deliverable

- **Usage guidance** (first-class, not just stories): when to use tooltip vs toggletip vs popover;
  content-length limits (tooltip ≤~80 chars, plain, supplementary/non-essential — never load-bearing
  info, since it's touch-dead and degrades to toggletip); do-not-nest rules; `tone`-means-status +
  icon convention; the "tooltip → toggletip on touch" degradation.
- **Storybook matrix** with addon-a11y per story: 12-placement grid, flip-at-edge, dark mode,
  forced-colors/HCM, long/scrolling content, delay-group demo, reduced-motion, disabled-trigger,
  nested overlay, controlled-open, all five `tone`s.
