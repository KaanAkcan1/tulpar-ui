# Tulpar UI v0.12 — Toast + Message (transient feedback family)

**Status:** Draft — pending spec review
**Date:** 2026-06-18
**Predecessor:** v0.11 Overlay Family (tooltip/toggletip/popover)
**Type:** New transient-feedback family + shared imperative service (core) + tokens + Angular/Vue wrappers

---

## 1. Overview

A transient, code-triggered feedback family driven by a single **imperative service**. Two public
channels split by weight and intent:

| Channel | Trigger | Placement | Content | Auto-dismiss | Stacks | ARIA |
|---|---|---|---|---|---|---|
| **Toast** | `toast.*()` | corner (default `bottom-right`) | icon · title · description · action(s) · close | 5s (configurable; danger/actionable persist by author choice) | yes | `status` / `alert` from intent |
| **Message** | `message.*()` | `top-center` | icon · single line | 3s | merges duplicates (grouping) | `status` |

Both are powered by **one framework-agnostic service** plus a single auto-mounted region element
(`<tulpar-toaster>`) that owns queueing, per-location stacking, timers, and the top-layer/portal.
This mirrors the input family's shared-internal pattern ([[project_input_architecture_decision]]):
one engine, multiple public entry points.

> **Dialog is explicitly NOT in this family.** A modal `<tulpar-dialog>` (focus-trap, scroll-lock,
> inert background, `confirm/alert/prompt`) is a separate, later spec. See §13 Out of Scope.

### Goals
- One correct, accessible, on-brand transient-feedback layer powering Toast + Message.
- An ergonomic imperative API (`toast.success(...)`, `toast.promise(...)`) that works from any code,
  not just components — exposed idiomatically per framework (Angular DI/signals, Vue composable).
- WCAG 2.2 AA, with the auto-dismiss timing tension (2.2.1) resolved by construction (§5).
- Brand-palette-aware tone, including a `custom` escape hatch that surfaces any brand family.

### Non-Goals (binding)
See §13 Out of Scope.

## 2. Architecture

```
@tulpar-ui/core
  src/_internal/toaster/        ← package-private region/queue/stack/timer engine + portal
  src/toast/tulpar-toast.ts     ← public custom element (declarative form + rich slot)
  src/toast/toast-service.ts    ← framework-agnostic imperative API (toast / message)
@tulpar-ui/tokens
  src/semantic/feedback.ts       ← new semantic feedback tokens (tone surfaces, motion, z, sizing)
@tulpar-ui/angular  → TulparToast / TulparMessage injectables (+ optional <tulpar-toast-ng> element)
@tulpar-ui/vue      → useTulparToast() / useTulparMessage() composables (+ <TulparToast> element)
```

- The toaster engine is consumed only within `@tulpar-ui/core` (the `_internal` boundary rule
  applies — never re-exported, never imported across packages).
- **`<tulpar-toaster>`** is a singleton region host. It is auto-created and appended to a defined
  portal root on the first service call if the app did not mount one explicitly. Mounting it
  explicitly (once, at app root) lets the consumer control the portal location and global defaults.
- Every `toast()`/`message()` call carries all of its own options (location, tone, duration, …);
  there is **no** per-location component to pre-declare (improves on PrimeVue's per-position
  `<Toast group>` model — one toaster manages all six locations).

### Units & boundaries
- **toaster engine** (`_internal`): owns a per-location queue, stacking math, timer controller
  (pause/resume), portal root, and theme-context re-establishment. Public surface: `enqueue`,
  `update`, `dismiss`, `dismissAll`, `setDefaults`. Independently testable with no framework.
- **toast-service**: thin object exposing `toast`/`message` + sugar methods; translates calls into
  engine `enqueue`/`update`/`dismiss`. No DOM logic of its own.
- **`<tulpar-toast>` element**: renders a single toast's visuals (tone surface, icon, title/desc
  slots, actions, close, timer ring, swipe). Pure presentation + interaction; the engine positions
  and times it. Usable declaratively on its own (controlled `open`) for the rich-content case.
- **wrappers**: expose the service as a DI injectable (Angular) / composable (Vue); forward the
  element's slots for the declarative/rich path.

## 3. Imperative service API

```ts
// every call returns a string id
toast(message: string, opts?: ToastOptions): string
toast.success | .info | .warning | .danger (message, opts?): string
toast.custom(message: string | HTMLElement, opts?: ToastOptions): string
toast.promise<T>(p: Promise<T>, msgs: {
  loading: string,
  success: string | ((data: T) => string),
  error:   string | ((err: unknown) => string),
}, opts?): string
toast.update(id: string, opts: Partial<ToastOptions>): void
toast.dismiss(id?: string): void          // id → that toast; omitted → all

message(text: string, opts?: MessageOptions): string
message.success | .info | .warning | .danger (text, opts?): string
message.dismiss(id?: string): void
```

```ts
interface ToastOptions {
  tone?: 'info' | 'success' | 'warning' | 'danger' | 'custom';   // default 'info'
  // custom-tone color: brand family name OR raw CSS color; plus part overrides
  color?: 'gok'|'ulgen'|'kizagan'|'erlik'|'ilay'|'umay' | string;
  bg?: string; accent?: string; text?: string;                   // part overrides
  highContrast?: boolean;                                         // danger escalation; no-op for non-danger tones
  title?: string;          // alias of the positional `message`
  description?: string;
  icon?: string | false;   // built-in name | raw SVG string | emoji; false = no icon
  action?: ToastAction | ToastAction[];
  closable?: boolean;      // default true; when true the × is always visible
  timer?: boolean;         // default true; false = persistent (no auto-dismiss, no ring)
  duration?: number;       // ms; default 5000 (Message 3000); 0 = persistent
  timerStyle?: 'track' | 'soft';   // default 'track'
  location?: Location;     // default toaster's global default ('bottom-right'); 6 values
  id?: string;             // supply to update/dedupe an existing toast
  onDismiss?: (reason: DismissReason) => void;
  onAutoClose?: () => void;           // fires specifically on timeout expiry
}
type ToastAction = { label: string; onClick: () => void };
type DismissReason = 'timeout' | 'user' | 'action' | 'programmatic' | 'swipe';
type Location = 'top-left'|'top-center'|'top-right'|'bottom-left'|'bottom-center'|'bottom-right';

interface MessageOptions {
  tone?: 'info'|'success'|'warning'|'danger';
  icon?: string | false;
  duration?: number;       // default 3000
  group?: boolean;         // default true; merge duplicate (tone+text) into a ×N count
  location?: Location;     // default 'top-center'
  id?: string;
}

// global defaults set once on the toaster (mount-time or setDefaults)
interface ToasterDefaults {
  location?: Location;     // default toast location ('bottom-right'); Message stays 'top-center'
  maxVisible?: number;     // default 3 — beyond this, toasts queue
  expand?: boolean;        // default false — true = always-expanded list (no collapse)
  duration?: number;       // global default toast duration (5000)
  timerStyle?: 'track' | 'soft';
}
```

- **Message is deliberately minimal:** auto-dismiss + grouping only. No actions, no close button, no
  lifecycle callbacks — that richness is what Toast is for. (A Message that needs an action is a
  Toast.)
- **`toast.promise` returns the original promise** (rethrows on rejection) so callers may
  `await`/`catch` it; the toast update is a side effect, never a swallow.
- **`onAutoClose` is convenience sugar** for the common "timeout expired" case; `onDismiss` with
  `reason:'timeout'` covers the same event for callers that want a single handler. Both fire on
  timeout (onAutoClose first).

- **Method name is `danger`** (not `error`) — matches the `tone` prop and the rest of the system.
- **`toast.custom`** accepts a string (rendered as a normal toast with `tone:'custom'`) **or** an
  `HTMLElement` (rendered into the toast's default body slot — the rich/arbitrary-content path).
- **`toast.promise`**: shows the `loading` toast (spinner icon, persistent, not closable) and
  `update()`s it to `success`/`error` on settle. The `success`/`error` values may be functions
  receiving the resolved value / rejection reason (Sonner model).
- **Persistence is author-controlled at call time** (`timer:false` / `duration:0`). The library does
  not silently force per-severity persistence; docs strongly recommend errors and actionable toasts
  be persistent (WCAG 2.2.1, §5).

## 4. Visual design

### 4.1 Tone (filled tonal surface)
The whole card is tinted by tone (Material/Fluent register), not merely a colored icon or border.

- **Built-in tones:** `info | success | warning | danger`. Each maps to a semantic feedback token
  set bound in the brand layer; **light** = light tinted surface + dark on-surface text + tonal
  border; **dark** = deep tinted surface (family toward 900/950) + lightened text/icon (100/200) +
  mid border, with a **1px border** carrying separation (shadows read weakly on dark).
- **`danger` high-contrast** (opt-in `highContrast:true`): saturated danger surface + white text,
  for genuinely urgent failures (Carbon high-contrast notification). Default danger stays tonal.
- **`custom` tone:** `color` accepts either
  - a **brand family name** (`gok·ulgen·kizagan·erlik·ilay·umay`) → resolved through the family's
    token scale with the same mode-aware light/dark flip as built-in tones (contrast-guaranteed), or
  - a **raw CSS color** (`#0d9488`, `rgb(...)`) → surface/border/text/ring derived from the single
    color (escape hatch; contrast is the author's responsibility).
  Plus **part overrides** (`bg` / `accent` / `text`, or the `--tulpar-toast-*` CSS variables) layered
  on top. `custom` is **visual-only** — it never changes `role`/`aria-live`/focus semantics.

### 4.2 Anatomy & density
`icon · title · description · action(s) · close`. Toast width fixed **360px** desktop, full-bleed
minus 16px on mobile; padding 12px 16px; title 14px/600, description 13px/400 muted; icon optically
aligned to the title baseline. Message is a pill, single line, auto-width (max ~480px), 8px 14px.

### 4.3 Icons (prop + slot, no bundled icon set)
1. **Default** built-in SVG per tone (info/success/warning/danger) when nothing is provided.
2. **`icon` prop:** built-in name, raw SVG string, or emoji; `icon: false` removes it.
3. **`slot="icon"`** (declarative element): arbitrary content — a Lucide component (Vue/Angular),
   custom `<svg>`, `<img>`. Tulpar ships only the four built-in tone icons; it bundles no icon set.

### 4.4 Timer indicator (perimeter ring)
The auto-dismiss countdown is a line that **depletes around the whole card perimeter** (SVG rect on
the border, `pathLength=100`, `stroke-dashoffset` 0→100), not a bottom bar.

- **`timerStyle: 'track'` (default):** a very faint static full-perimeter track under a gentle
  depleting fill (~1.5px, ~0.7 opacity).
- **`timerStyle: 'soft'`:** depleting fill only (~1.5px, ~0.6 opacity), no track.
- Hidden when `timer:false` or the toast is persistent. Color = tone accent. Pauses on hover **and**
  keyboard focus (§5). Under `prefers-reduced-motion` the ring does not animate (static border).

### 4.5 Title / description prop + slot duality
Per [[feedback_optional_prop_does_not_remove_slot]]: `title` and `description` are available as both
the imperative string options **and** named slots (`slot="title"`, `slot="description"`) on the
declarative element. The slot wins when both are present.

## 5. Timing, dismissal & accessibility

### 5.1 Durations & persistence
- Default **Toast 5s**, **Message 3s**; overridable per call via `duration`.
- **Pause on hover AND on keyboard focus**; resume (from remaining time) on leave/blur. Satisfies the
  WCAG 2.2.1 "extend / turn off" intent.
- Persistent = `timer:false` / `duration:0`. Docs recommend errors and actionable toasts be
  persistent so information/actions are not missed (Atlassian/Carbon/MUI consensus, WCAG 2.2.1).

### 5.2 WCAG 2.2.1 stance (documented)
Auto-dismiss is conformant only because the same information is non-critical / verifiable elsewhere
(W3C's own toast example). Therefore: auto-dismiss is for redundant confirmations (Message, success);
anything non-redundant or actionable should be made persistent by the author. A manual close
affordance is always available (`closable` default true).

### 5.3 ARIA contract
- **Message** → `role="status"`, `aria-live="polite"`; never focusable; never steals focus.
- **Toast** → politeness derived from intent: `info`/`success`/`warning` → `status`/polite;
  `danger` → `alert`/assertive. The toast **never steals focus** on appearance.
- **Actionable toast** (has an `action`) → escalates to `alertdialog` semantics inside a labelled
  landmark region `<… role="region" aria-label="Notifications">` (React Aria + Carbon), because
  focusable content inside a polite live region is an a11y trap.
- **Keyboard:** an **F6 / Shift+F6** hotkey jumps focus to the toast region from anywhere; **Esc**
  dismisses the focused toast; on dismiss, focus moves to the next toast, then is restored when the
  region empties (React Aria). The F6 hotkey is documented (discoverability is on us).
- **Region placement:** the toaster region renders at document top level, **never inside a dialog's
  modal subtree** (a live region inside `aria-modal` is not announced — known MUI bug).
- **forced-colors / HCM:** every toast/message keeps a `1px solid` `CanvasText` border so it survives
  shadow stripping; tone is conveyed by border + text, never icon color alone; `tone`/`custom` tint
  becomes a no-op (author colors overridden), text label persists.

### 5.4 Swipe-to-dismiss
Pointer/touch horizontal swipe past a threshold (or with sufficient velocity) dismisses with
`reason:'swipe'`; below threshold it springs back. Disabled under `prefers-reduced-motion`. Keyboard
and the × button remain the primary, always-available dismiss paths.

## 6. Stacking, location & motion

### 6.1 Stacking
- **Default (Sonner):** collapsed stack shows **3** visible toasts; toasts behind scale down
  (`1 − 0.05·i`) and lift (`−14px·i`); overflow stays queued. Hovering/focusing the stack **expands**
  it vertically (real heights + gap) **and pauses all timers**. (Per-toast focus-pause in §5.1
  composes with this: focusing one toast expands the stack and therefore pauses the whole stack.)
- **`expand` (toaster option):** always-expanded list (no collapse) — Chakra `overlap:false` / Sonner
  `expand`.
- **Max visible** is configurable on the toaster (default 3); beyond it, toasts queue.
- Each location maintains its own independent stack; top locations grow downward, bottom locations
  grow upward.

### 6.2 Location
- **Six locations:** `top-left | top-center | top-right | bottom-left | bottom-center | bottom-right`.
- Toaster has a **global default** (`bottom-right` for Toast; Message is pinned `top-center`); each
  call may **override** via `location`.

### 6.3 Motion (CSS transitions, not keyframes — interruptible)
- **Enter:** opacity 0→1 + `translateY(16px)→0` + `scale(0.96)→1`, ~220ms `cubic-bezier(.22,1,.36,1)`.
- **Exit:** opacity→0 + height collapse, ~160ms (exit shorter than enter).
- **Stack reposition:** `transform`/`top` transition ~300ms, same ease, so cards glide when one
  leaves.
- **Message enter:** `translateY(-12px)→0` + fade, ~180ms (lighter/faster).
- All durations/easings are **motion tokens**; `prefers-reduced-motion` drops transform/scale to a
  ≤100ms opacity fade (stack still repositions, but instantly).

## 7. Tokens (new — `@tulpar-ui/tokens` semantic/feedback.ts)

Bound in the brand layer (`brand/tulpar/{light,dark}.ts`); component CSS consumes **semantic tokens
only** (ESLint-enforced; provide `var()` fallbacks).

- `--tulpar-color-feedback-{info,success,warning,danger}-{surface,on-surface,border,accent}` ×
  light/dark (the tonal surface model).
- `--tulpar-color-feedback-danger-hc-{surface,on}` — the high-contrast danger escalation.
- A dedicated **overlay/toast shadow ramp** (elevation distinct from cards).
- `--tulpar-z-toast` — top-layer/portal z-tier; must sit above shell/sidenav and overlay family.
- Motion: `--tulpar-toast-duration-{enter,exit,reposition}`, `--tulpar-toast-easing`,
  `--tulpar-toast-delay-default` (5000 toast / 3000 message defaults live as tokens).
- Sizing: toast `max-width` (~360px) + padding scale; message pill padding; timer-ring stroke widths.
- Public **CSS custom-property hooks** for the `custom` tone part-overrides: `--tulpar-toast-bg`,
  `--tulpar-toast-accent`, `--tulpar-toast-text`.

The `custom` brand-family resolution reuses the existing `primitive/color.ts` family scales
(`gok·ulgen·kizagan·erlik·ilay·umay`); the public API never exposes primitive token spelling beyond
the family name.

## 8. Wrappers

Both wrappers expose the **identical service surface**; only the access idiom differs.

### Angular (signals, no RxJS — [[feedback_angular_signals_only]])
```ts
const toast = inject(TulparToast);
toast.success('Kaydedildi');
const ok = toast.promise(save(), { loading:'Kaydediliyor…', success:(r)=>`${r.count} öğe`, error:'Hata' });
```
- `provideTulparToast()` (`providedIn:'root'`). Queue state exposed as a `signal()`/`computed()` for
  anyone rendering a custom region. `toast.promise` returns/await via Promise — **no RxJS**.
- Optional `<tulpar-toast-ng>` element wrapper for the declarative/rich-content path; forwards
  `slot="title|description|icon"` and the default slot.

### Vue (composables)
```ts
const toast = useTulparToast();
toast.danger('Yükleme başarısız', { action: { label:'Tekrar dene', onClick: retry } });
```
- `useTulparToast()` / `useTulparMessage()` return the service bound to the app instance
  (Element Plus / Ant-Vue model). `app.use(TulparToastPlugin)` mounts the toaster once.
- The `<TulparToast>` SFC forwards named slots through transparent `display:contents` carriers
  (the Vue named-slot forwarding gotcha in CLAUDE.md applies).

## 13. Out of Scope (binding)

- **Dialog / modal** — `<tulpar-dialog>` (focus-trap, scroll-lock, inert background,
  `confirm/alert/prompt`, mobile bottom-sheet) is a **separate, later spec**. Parked here
  deliberately; a reference mockup exists but no contract is implied.
- **`important` / priority queue** — deferred; `aria-live=assertive` on danger already covers urgent
  announcement. Additive later; the queue API must not foreclose it.
- **Notification center / history / persistence across reloads** — future.
- **Multiple-trigger → one-content**, **virtualized very long toast queues** — deferred.
- **RTL verification** — the engine is logical-direction-aware where cheap, but RTL testing is
  project-wide future scope.
- **Bundled icon set** — Tulpar ships only the four built-in tone icons; consumers bring their own
  (Lucide, etc.) via the icon slot/prop.
- **React wrapper, SSR tuning, mobile-native** — out (project-wide).
