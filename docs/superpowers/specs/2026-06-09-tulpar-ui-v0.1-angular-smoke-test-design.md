# Tulpar UI v0.1 ‚Äî Angular Published Package Smoke Test

**Date:** 2026-06-09
**Owner:** Kaan Akcan
**Status:** Spec ‚Äî pre-implementation

## Goal

Verify that `@tulpar-ui/angular@0.1.0` (just published to the public npm registry) is installable and usable by a real Angular consumer who is **outside the monorepo** and uses plain `npm install`.

Success means a fresh `ng new` project can `npm i @tulpar-ui/angular`, import the wrapper, and render a working `<tulpar-button-ng>` in the browser.

## Background

All three packages were published to npmjs.org six minutes before this spec was written:

- `@tulpar-ui/tokens@0.1.0`
- `@tulpar-ui/core@0.1.0` ‚Äî peer dep on `tokens@0.1.0`
- `@tulpar-ui/angular@0.1.0` ‚Äî peer deps on `@angular/common@^22`, `@angular/core@^22`, `lucide-angular>=0.500.0`, `core@0.1.0`

`pnpm publish` correctly substituted `workspace:*` with concrete versions, so the manifests on the registry are clean.

Until now the only Angular consumer was `apps/playground-ng`, which uses `workspace:*` and bypasses the npm install path entirely. The published install path has never been exercised.

## Location

`C:\Source\Kaan\tulpar-ui-test-project` ‚Äî outside the monorepo so there is no workspace symlink bleed-through. This is the directory the user will keep around as a public-install sanity harness for future releases.

## Approach

Use the canonical Angular CLI flow (`ng new`) at the version that matches our peer dep range. Anything else (hand-rolled Vite + Angular, copying playground-ng) would either not exercise CLI integration or would re-use workspace links, defeating the purpose.

## Steps

1. **Scaffold** ‚Äî `npx -p @angular/cli@22 ng new tulpar-ui-test-project --standalone --routing=false --style=css --skip-git --skip-tests` from `C:\Source\Kaan`.
2. **Install** ‚Äî inside the new project: `npm i @tulpar-ui/angular lucide-angular`. npm 7+ peer auto-install will pull `@tulpar-ui/core@0.1.0` and `@tulpar-ui/tokens@0.1.0`.
3. **Load tokens CSS** ‚Äî prepend `@import "@tulpar-ui/tokens/css/tulpar.css";` to `src/styles.css`.
4. **Register the Web Component** ‚Äî add side-effect import `import "@tulpar-ui/core/button";` to `src/main.ts` so `customElements.define("tulpar-button", ...)` runs before Angular bootstraps.
5. **Wire the wrapper** ‚Äî in `src/app/app.ts`:
   - `import { TulparButtonComponent } from "@tulpar-ui/angular";`
   - Add it to the standalone component's `imports: [...]`.
6. **Template** ‚Äî `src/app/app.html`:
   ```html
   <tulpar-button-ng variant="primary" (clicked)="onClick('primary')">Primary</tulpar-button-ng>
   <tulpar-button-ng variant="secondary" (clicked)="onClick('secondary')">Secondary</tulpar-button-ng>
   <tulpar-button-ng variant="ghost" disabled>Disabled</tulpar-button-ng>
   ```
7. **Click handler** ‚Äî `onClick(variant: string) { console.log("clicked", variant); }`.
8. **Run** ‚Äî `ng serve`, open browser, verify visually and via console.

## Acceptance Criteria

- `npm install` completes with no `EUNSUPPORTEDPROTOCOL` (`workspace:`) errors and no unresolved peers beyond what is documented.
- `ng serve` boots cleanly ‚Äî no template-parser, schema, or TS errors.
- Three buttons render in the browser:
  - Primary: brand-color filled
  - Secondary: outlined / neutral
  - Ghost (disabled): transparent and not interactive
- Clicking primary or secondary logs `"clicked primary"` / `"clicked secondary"` to the console.
- Clicking the disabled button does **not** log anything.
- DevTools shows `<tulpar-button>` inside `<tulpar-button-ng>` with a real Shadow DOM tree.

## Out of Scope (deliberate)

- ButtonGroup smoke (separate follow-up if base button passes)
- Dark mode toggle
- Form-associated submit behavior inside `<form>`
- Lucide icon rendering via `icon` prop
- Vue package smoke test (separate harness)
- E2E automation (Playwright / Cypress) ‚Äî manual verification suffices for v0.1
- CI integration of this smoke test

## Known Risk Points

These are not pre-decided fixes; they are things to watch for during execution. If any of them fires, treat it as a real bug in the published package and surface it before patching the test project.

- **`useDefineForClassFields`** ‚Äî Lit needs `false`, Angular 22's default `tsconfig` may set `true`. The wrapper should encapsulate this, but unverified for the published artifact.
- **`lucide-angular` peer dep tension** ‚Äî Angular 22's CLI may complain about Lucide's peer range. We declared it as a peer at `>=0.500.0`; npm should resolve cleanly, but worth watching.
- **`@tulpar-ui/core/button` side-effect import** ‚Äî depends on the published exports map registering on import. If the published `dist/button/index.js` doesn't auto-register, we'd need an explicit `register()` call and the wrapper docs would need to say so.
- **Angular schema warnings for unknown elements** ‚Äî the wrapper should expose `CUSTOM_ELEMENTS_SCHEMA` internally so the consumer doesn't need to. If a schema error surfaces, that's a wrapper bug.
