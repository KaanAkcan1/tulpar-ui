# Tulpar UI v0.1.1 — Install UX Patch (lucide decoupling + peer→dep)

**Date:** 2026-06-09
**Owner:** Kaan Akcan
**Status:** Spec — pre-implementation
**Predecessor:** [v0.1.0 published 2026-06-09](2026-06-05-tulpar-ui-v0.1-button-design.md), smoke tested in [test project spec](2026-06-09-tulpar-ui-v0.1-angular-smoke-test-design.md)

## Why

The v0.1 smoke test in `C:\Source\Kaan\tulpar-ui-test-project` surfaced three real defects in the published `@tulpar-ui/angular@0.1.0` install path. None blocks the Web Component itself; all three break the "fresh `ng new` → `npm i @tulpar-ui/angular`" promise.

### Defect 1 — peer dependencies don't auto-install on `--legacy-peer-deps`

`@tulpar-ui/angular` declares `@tulpar-ui/core@0.1.0` as a peer. Same for `@tulpar-ui/core` → `@tulpar-ui/tokens`. With npm peer auto-install enabled (npm 7+ default), this works. But because of Defect 2 below, consumers MUST pass `--legacy-peer-deps`, which disables peer auto-install. They then get `@tulpar-ui/angular` alone — `core` and `tokens` go missing, runtime fails.

### Defect 2 — lucide-angular hard-imported, blocks Angular 22 install

`packages/angular/src/button/tulpar-button.component.ts:13` hard-imports `LucideAngularModule`. The wrapper declares `lucide-angular>=0.500.0` as a peer. But every published version of `lucide-angular` (and the renamed `@lucide/angular`) caps at `@angular/common: 13.x - 21.x`. **There is no lucide release that supports Angular 22.** Any fresh `ng new` on Angular 22 hits `ERESOLVE` immediately.

Even if the user passes `--legacy-peer-deps`, the wrapper still imports lucide at build time, forcing the consumer to install lucide even if they never use icons.

### Defect 3 — wrong icon API for Angular

Vue's `icon: Component` prop works because lucide-vue-next provides icons as Vue components — `Check` is a component, `:icon="Check"` is zero-cost. Angular's lucide-angular provides icons as `LucideIconData` (path data, not a component); rendering requires `<lucide-angular [img]="Check">`. So the Angular wrapper hard-couples to lucide-angular as the rendering library, which is the wrong layer to make that choice.

## Goal

A consumer running `npm install @tulpar-ui/angular` from a fresh Angular 22 app:

1. Install succeeds with **no** `--legacy-peer-deps` flag.
2. No `lucide-angular` installed automatically (consumer adds it only if they want to use lucide).
3. `core` and `tokens` come along with `angular` automatically.
4. Buttons without icons work out of the box.
5. Buttons with icons work via either of two patterns the consumer chooses (see below).

## Design — Option A: icon prop accepts a generic component, slot still works

Mirror Vue's pattern. The `icon` input becomes `Type<unknown> | undefined`. The wrapper renders the dynamic component via `*ngComponentOutlet` and passes `{ size: effectiveIconSize() }` as an input. Lucide is no longer touched by the wrapper.

**Two consumer patterns, both supported by the same wrapper:**

### Pattern 1 — slot directly (no boilerplate)

```ts
import { Component } from "@angular/core";
import { TulparButtonComponent } from "@tulpar-ui/angular";
import { LucideAngularModule, Check } from "lucide-angular";

@Component({
  imports: [TulparButtonComponent, LucideAngularModule],
  template: `
    <tulpar-button-ng>
      <lucide-angular slot="start" [img]="Check" [size]="16"></lucide-angular>
      Save
    </tulpar-button-ng>
  `,
})
export class SaveButton {
  readonly Check = Check;
}
```

### Pattern 2 — named icon component via `[icon]` prop

```ts
// app-check-icon.ts (5-line wrapper, reusable)
@Component({
  selector: "app-check-icon",
  standalone: true,
  imports: [LucideAngularModule],
  template: `<lucide-angular [img]="iconData" [size]="size()"></lucide-angular>`,
})
export class AppCheckIcon {
  readonly iconData = Check;
  size = input(16);
}
```

```html
<tulpar-button-ng [icon]="AppCheckIcon">Save</tulpar-button-ng>
```

Both patterns coexist because the wrapper template conditionally renders the prop and always renders the slot:

```html
<tulpar-button ...>
  @if (icon()) {
    <span slot="start">
      <ng-container *ngComponentOutlet="icon(); inputs: { size: effectiveIconSize() }" />
    </span>
  }
  <ng-content select="[slot=start]" />
  <ng-content />
  <ng-content select="[slot=end]" />
  <ng-content select="[slot=loading-icon]" />
</tulpar-button>
```

This is a strict superset of slot-only — providing the prop adds no cost when unused, removing the prop would lock out Pattern 2 unnecessarily.

## Changes

### `packages/angular/src/button/tulpar-button.component.ts`

- Remove `import { LucideAngularModule, type LucideIconData } from "lucide-angular"`.
- Add `import { type Type } from "@angular/core"` (`NgComponentOutlet` from `@angular/common`).
- Change `imports` array from `[LucideAngularModule]` to `[NgComponentOutlet]`.
- Change `icon` input type: `LucideIconData | undefined` → `Type<unknown> | undefined`.
- Replace `<lucide-angular [img]="icon()" [size]="effectiveIconSize()"></lucide-angular>` with `<ng-container *ngComponentOutlet="icon(); inputs: { size: effectiveIconSize() }" />`.
- Keep `effectiveIconSize`, `autoIconOnly`, `hasText`, `contentWrapper` — all still valid.

### `packages/angular/package.json`

- Remove `lucide-angular` from `peerDependencies` and `devDependencies`.
- Move `@tulpar-ui/core` from `peerDependencies` → `dependencies` (keep version `workspace:*` for source; `pnpm publish` resolves it).
- Add `@tulpar-ui/tokens` to `dependencies` (currently transitive through core's peer; make explicit because consumer needs the CSS).
- Bump version → `0.1.1`.

### `packages/core/package.json`

- Move `@tulpar-ui/tokens` from `peerDependencies` → `dependencies`.
- Bump version → `0.1.1`.

### `packages/vue/package.json`

- Move `@tulpar-ui/core` from `peerDependencies` → `dependencies` (consistency with angular package; no functional change for current Vue consumers).
- Bump version → `0.1.1`.

### `packages/tokens/package.json`

- Bump version → `0.1.1` (carried along by changesets even though no source change, so consumers can pin a coherent set).

### `apps/playground-ng/src/app/app.ts`

- Lucide is still legitimately used here for demonstration. Migrate the existing `[icon]="Check"` style demos to one of two patterns:
  - For most demos, switch to **Pattern 1** (slot directly with `<lucide-angular slot="start" [img]="Check">`). Cleaner, mirrors what most consumers will write.
  - Add at least one demo using **Pattern 2** (named component) to document that pattern.
- Existing `lucide-angular` devDependency on the playground stays — playground keeps using lucide directly, the change is in HOW it uses it.

### `packages/angular/src/button/tulpar-button.component.spec.ts`

- Existing tests don't touch the `icon` prop, but add at least:
  - One test passing a dummy `Type<unknown>` component to `icon` input and asserting it renders inside `<span slot="start">`.
  - One test passing no `icon` input and projecting content into `slot=start` — asserting projection still works.

## Acceptance Criteria (verified via `C:\Source\Kaan\tulpar-ui-test-project`)

1. Delete the test project's `node_modules` and `package-lock.json`.
2. `npm install @tulpar-ui/angular` (no flags) succeeds. `node_modules/@tulpar-ui/` contains `tokens`, `core`, `angular`. No `lucide-angular` anywhere in the tree.
3. `ng serve` boots cleanly.
4. Buttons render (no template/schema errors from `*ngComponentOutlet`).
5. Click event still fires; disabled still blocks.
6. As an extra verification, install `lucide-angular --legacy-peer-deps` (upstream still doesn't support Angular 22), then add Pattern 1 and Pattern 2 demos to the test app; both render.

## Out of Scope

- `lucide-angular` upstream Angular 22 support — we cannot fix that.
- React/Solid/Svelte wrappers — v0.1 only ships Angular + Vue.
- Component renaming, additional button props, or any feature beyond what fixes the three defects.
- Repackaging the `lucide-angular` rename to `@lucide/angular`. We're not using either anymore; consumer picks.

## Versioning Note

Strict pre-1.0 semver says removing/retyping a public input is a 0.2.0 (breaking) bump. We are calling this 0.1.1 because:
- v0.1.0 was published one hour before this patch.
- Zero external consumers exist.
- The smoke test project is the only known consumer and is co-owned with this work.

The Changeset file should still record this as a **breaking change** for the angular package, so the changelog reflects truth even if the version number doesn't enforce it.

## Release Flow

1. Run all changes locally.
2. `pnpm build` and `pnpm test` clean.
3. Local smoke test in `tulpar-ui-test-project` against the BUILT packages (`pnpm pack` + `npm install ./tgz`) before publishing.
4. `pnpm changeset` — one entry per affected package with the appropriate bump (patch for angular as user policy, even though strict semver suggests minor).
5. `pnpm version-packages` + `pnpm release` to publish to npm.
6. Final smoke test in `tulpar-ui-test-project` against the published `@tulpar-ui/angular@0.1.1` to confirm acceptance criteria.
