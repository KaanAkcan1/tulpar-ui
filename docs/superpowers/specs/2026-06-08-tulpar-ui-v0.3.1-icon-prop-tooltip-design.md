# Tulpar UI v0.3.1 — `:icon` prop + `tooltip` attribute

**Status:** Approved 2026-06-08
**Predecessor:** [v0.3 — Button API expansion](./2026-06-08-tulpar-ui-v0.3-button-api-expansion-design.md)
**Type:** Additive patch (no breaking changes — existing slot-based icon API stays as fallback)

---

## 1. Overview

Two additive enhancements on top of v0.3 Button:

1. **`:icon` + `:icon-size` props** on the Vue and Angular wrappers — a convenience layer for Lucide icons that replaces verbose `<span slot="start">` boilerplate. Existing slot pattern stays as escape hatch for non-Lucide icon libraries (Heroicons, Tabler, Material, custom SVG).
2. **`tooltip` attribute** on the core Button — string-only inline tooltip shown on hover/focus. Auto-fills the a11y gap for icon-only buttons. Rich/component popovers remain out of scope (future `<tulpar-popover>`).

Plus one bug fix:
- **Playground dev-server cache** — `.vite/deps` and `.angular/cache` cache stale token CSS, hiding token updates. Add Vite `optimizeDeps.exclude` for `@tulpar-ui/tokens`; document Angular cache clear procedure.

## 2. Goals & Non-Goals

**Goals:**
- Reduce icon usage boilerplate at the wrapper level for the most common case (Lucide).
- Provide a built-in, accessible tooltip for icon-only buttons without forcing a separate Popover dependency.
- Preserve library-agnostic core (slot-based) for consumers using non-Lucide icon sets.

**Non-Goals (binding):**
- No rich/component popovers — only string tooltip. Rich popovers = future `<tulpar-popover>`.
- No collision detection / smart positioning for tooltip — fixed below the button.
- No icon library bundling — Tulpar still does not ship icons. Consumer installs `lucide-vue-next` / `lucide-angular` themselves.
- No multi-icon support via `:icon` — single icon only. Multi-icon use case (`[← Back to ... →]`) keeps using the slot pattern.

## 3. `:icon` prop (Vue + Angular wrappers)

### 3.1 API

**Vue:**
```ts
interface Props {
  icon?: Component;          // Lucide-vue-next component reference
  iconSize?: number;         // Optional override; default derived from `size` attribute
  // ... v0.3 props unchanged
}
```

**Angular:**
```ts
icon = input<LucideIconData | undefined>(undefined);
iconSize = input<number | undefined>(undefined);
// ... v0.3 inputs unchanged
```

### 3.2 Behavior

- When `icon` is set, the wrapper internally renders the icon inside a `<span slot="start">` so it lands in the core Web Component's start slot.
- `iconSize` defaults to the per-size mapping: xs=12, sm=14, md=16, lg=18, xl=20, 2xl=24, 3xl=28. Consumer-set `iconSize` overrides this.
- `icon-position` attribute continues to control flex direction (start/end/top/bottom), so the same single icon can be positioned anywhere via that prop. `:icon` alone always renders in the start slot.

### 3.3 Slot-based fallback (preserved)

For non-Lucide consumers:

```vue
<TulparButton severity="primary">
  <span slot="start"><HeroIcon :class="'w-4 h-4'" /></span>
  Save
</TulparButton>
```

Both `:icon` and `<span slot="start">` can co-exist — if both set, `:icon` wins (renders first), slotted content joins at appropriate position.

### 3.4 Auto icon-only

When `:icon` is set AND the default slot has no text content, the wrapper automatically applies `icon-only` mode (square padding, fixed width = height).

**Vue implementation** — straightforward via `useSlots()`:
```ts
import { useSlots, computed } from "vue";
const slots = useSlots();
const autoIconOnly = computed(() => Boolean(props.icon) && !slots.default);
```

**Angular implementation** — uses element textContent inspection after content init:
```ts
@ViewChild('contentSlot', { static: false }) contentRef!: ElementRef;
hasText = signal(false);

ngAfterContentInit() {
  const text = this.elementRef.nativeElement.textContent?.trim() ?? "";
  this.hasText.set(text.length > 0);
}

autoIconOnly = computed(() => Boolean(this.icon()) && !this.hasText());
```

Both wrappers expose the same consumer API: `<TulparButton :icon="X" aria-label="..." />` becomes icon-only automatically. Explicit `:icon-only="true"` still works and takes precedence.

A11y warning: if `auto icon-only` triggers without `aria-label`, the core Button already warns (existing behavior). No new logic.

## 4. `tooltip` attribute (core Web Component)

### 4.1 API

```ts
@property({ type: String, reflect: true })
tooltip?: string;
```

### 4.2 Behavior

- When `tooltip` is set, the core Button renders an additional `<span class="tooltip" role="tooltip" id="btn-tip">${tooltip}</span>` in shadow DOM.
- The inner `<button>` / `<a>` gets `aria-describedby="btn-tip"`.
- CSS shows the tooltip on `:hover` and `:focus-visible` of the button. Hidden by default.

### 4.3 Visual / position

- Fixed position: 8px below the button, centered horizontally.
- No collision detection for v0.3.1. Consumer responsible for layout. (Future `<tulpar-popover>` handles complex positioning.)
- Tooltip background: `var(--tulpar-color-text-primary)` (dark in light theme, light in dark theme).
- Tooltip text: `var(--tulpar-color-text-inverse)`.
- Transition: 150ms ease opacity.
- z-index: 100.

### 4.4 A11y

- `role="tooltip"` on the span.
- `aria-describedby` on the inner button/anchor.
- Show on `:hover` AND `:focus-visible` (keyboard users get tooltip too).
- Tooltip text is part of accessibility tree via aria-describedby.

## 5. Cache fix

### 5.1 Vite (playground-vue)

Add to `apps/playground-vue/vite.config.ts`:
```ts
export default defineConfig({
  ...
  optimizeDeps: {
    exclude: ["@tulpar-ui/tokens", "@tulpar-ui/core", "@tulpar-ui/vue"],
  },
});
```

Prevents Vite from pre-bundling Tulpar packages, so token CSS changes propagate immediately on dev server reload.

### 5.2 Angular (playground-ng)

Document procedure:
```
rm -rf apps/playground-ng/.angular/cache
pnpm --filter playground-ng start
```

Angular CLI doesn't easily support per-package cache exclusion. The `.angular/cache` is regenerated on each `ng serve`; clearing it forces a fresh build.

### 5.3 README

Add a "Dev server cache" troubleshooting section to README explaining when to clear caches (after `pnpm --filter @tulpar-ui/tokens build` while playground is running).

## 7. Success Criteria

- `<TulparButton :icon="Check">Save</TulparButton>` renders identically to `<TulparButton><span slot="start"><Check :size="16" /></span>Save</TulparButton>`.
- `<TulparButton :icon="Check" aria-label="Save" />` renders as icon-only (square, no text area).
- `<TulparButton tooltip="Delete">...</TulparButton>` shows tooltip on hover and focus, with proper aria-describedby.
- Existing slot-based pattern still works for Heroicons / Tabler / Material / custom SVG examples.
- Playground premium severity shows gold (`#b8985a`) after cache clear + dev server restart.
- All v0.3 tests still pass.
- Workspace `pnpm build`, `pnpm lint`, `pnpm format:check` all green.

## 8. Out of Scope

- Rich popover (component content) — future v0.4 `<tulpar-popover>`.
- Tooltip position attribute (`top`/`bottom`/`left`/`right`) — future iteration.
- Tooltip collision detection — future iteration.
- Icon library auto-discovery / registry — consumer brings their own.
- Multi-icon via `:icon` (only single icon supported by prop; multi-icon uses slot pattern).
