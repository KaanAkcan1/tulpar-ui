# Tulpar UI v0.7 — Button Enterprise Polish Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Elevate `<tulpar-button>` to an enterprise/product-ready visual finish (surface physics, motion personality, press feedback, focus clarity, semantic disabled, radius-scale, premium +1 light, spinner reflow-safety) with **zero new public API and zero breaking change**.

**Architecture:** Visual values flow through tokens. New semantic/brand tokens are added to `@tulpar-ui/tokens` (TS source → generated `--tulpar-*` CSS vars); all `color-mix` math lives in the brand layer. `@tulpar-ui/core`'s `tulpar-button.styles.ts` consumes only semantic/composite tokens (never `--tulpar-primitive-*`). Wrappers inherit the new feel for free.

**Tech Stack:** TypeScript token objects + custom CSS generator (`packages/tokens/build/generate-css.ts`), Lit 3 `css` templates, Web Test Runner + `@open-wc/testing` (core), Vitest (tokens), Storybook 8, Changesets.

**Spec:** [`docs/superpowers/specs/2026-06-16-tulpar-ui-v0.7-button-polish-design.md`](../specs/2026-06-16-tulpar-ui-v0.7-button-polish-design.md)

> Do **not** create the branch or run any command until the user explicitly approves execution.

**2. One architecture decision to confirm (shadow color source).** The colored shadow layers and the self-border must follow BOTH the `severity` stops AND the JS `color` attribute override (`_applyColorOverride()` in `tulpar-button.ts:211-230`, which sets the private `--_btn-color-default`). CSS custom properties resolve **at the use site**, so a token defined on `:root` whose value contains `var(--_btn-color-default)` resolves correctly when applied to `.btn` inside the shadow DOM.
> **Recommended (this plan assumes it):** brand-layer composite shadow/border tokens reference `var(--_btn-color-default)` internally. This keeps all `color-mix` in the brand layer (honors spec Decision 9), keeps component CSS trivial (`box-shadow: var(--tulpar-button-shadow-rest)`), and makes shadows follow severity + color override automatically. Accepted trade-off: the tokens package references the button's private accent var (a documented, button-specific bridge).
> **Alternative (Option B):** component owns the `color-mix` recipe over `--_btn-color-default` (allowed by ESLint — not a primitive). Pick this only if we object to tokens referencing `--_btn-*`.
> Confirm Option A vs B with the user before Task 2.

## Token name reference (generator output)

The generator (`flattenTokens`) maps camelCase TS paths to kebab `--tulpar-<path>`. Final names this plan introduces:

| TS path (in `SemanticTokens`) | Generated CSS var |
|---|---|
| `transition.easeStandard` | `--tulpar-transition-ease-standard` |
| `button.pressDuration` | `--tulpar-button-press-duration` |
| `button.spinnerDuration` | `--tulpar-button-spinner-duration` |
| `button.size.<s>.radius` | `--tulpar-button-size-<s>-radius` |
| `button.size.<s>.letterSpacing` | `--tulpar-button-size-<s>-letter-spacing` |
| `button.disabled.bg/fg/border` | `--tulpar-button-disabled-bg/-fg/-border` |
| `button.surfaceHighlight` | `--tulpar-button-surface-highlight` |
| `button.surfaceShade` | `--tulpar-button-surface-shade` |
| `button.surfaceBorder` | `--tulpar-button-surface-border` |
| `button.shadow.rest/hover/raised/press` | `--tulpar-button-shadow-rest/-hover/-raised/-press` |
| `button.premium.sheen/ambient` | `--tulpar-button-premium-sheen/-ambient` |

`color.focusRing` (`--tulpar-color-focus-ring`) already exists — reused, not added.

> **Spec name reconciliation:** spec §5 used `--tulpar-ease-standard` and `--tulpar-shadow-button-*`; the generator forces `--tulpar-transition-ease-standard` and `--tulpar-button-shadow-*`. The names in this table are authoritative. The existing component fallback reference `var(--tulpar-button-shadow-raised, …)` (`tulpar-button.styles.ts:300`) already matches `button.shadow.raised` — good, the fallback simply becomes a real token.

> **Disabled token naming (deliberate deviation from spec §5/Decision 6).** The spec named *semantic* tokens `--tulpar-color-{bg,fg,border}-disabled`. This plan instead scopes them to the button: `button.disabled.{bg,fg,border}` → `--tulpar-button-disabled-{bg,fg,border}`. Rationale: YAGNI — no other component needs a generic disabled color yet, and a button-scoped token keeps the global `ColorTokens` contract unchanged and the value independently tunable. Verified: nothing currently consumes `--tulpar-color-*-disabled`. If a second consumer appears later, promote to semantic then.

> **Color-override × color-mix caveat.** When the `color` attribute is set, `_applyColorOverride()` (`tulpar-button.ts:223`) binds `--_btn-color-default` to a `--tulpar-primitive-color-*` *var*. The brand-layer `color-mix(in oklch, var(--_btn-color-default) …)` then mixes a primitive **value** at runtime — this is fine and does NOT violate the primitive-token ESLint rule (that rule scans `*.styles.ts` source text only, never resolved values). Do not "fix" this later.

## Chunk 3: Loading reflow + reduced-motion

### Task 7: Width-preserving loading for start/end positions

**Files:**
- Modify: `packages/core/src/button/tulpar-button.styles.ts`
- Modify (only if needed): `packages/core/src/button/tulpar-button.ts`

> `center` already preserves width (`visibility:hidden` + absolute spinner). The gap is `start`/`end`, which use `display:none` on `.start`/`.end` and insert the spinner inline → width shifts.

- [ ] **Step 1: Write failing test (width stable across loading toggle)**

```ts
it("button width does not change when loading toggles (start position)", async () => {
  const el = await fixture<TulparButton>(
    html`<tulpar-button loading-position="start">Save changes</tulpar-button>`,
  );

  el.loading = true;
  await el.updateComplete;

});
```

- [ ] **Step 2: Run — expect FAIL** (width jumps)

Run: `pnpm --filter @tulpar-ui/core test`

- [ ] **Step 3: Change start/end loading to absolute overlay (preserves width)**

**Why the naive approach fails:** `.spinner` is a *sibling* flex item rendered after `.start`/`.label`/`.end` (`tulpar-button.ts:316`), NOT inside the icon slot. Setting `.start { visibility: hidden }` (keeps its box) **and** `.spinner { display: inline-flex }` (adds a box) makes width = slot-box + spinner-box → **width grows**. `center` mode avoids this by taking the spinner out of flow with `position: absolute` (`tulpar-button.styles.ts:440`). Mirror that for start/end.

`.btn` is already `position: relative` (`tulpar-button.styles.ts:146`), so the spinner can be absolutely positioned within it. Replace the existing `loading-position="start"/"end"` rules with:
```css
/* Loading — start/end: keep the icon slot's box (visibility:hidden so width
   is preserved), overlay the spinner at that edge via absolute positioning
   so it adds no inline box. */
:host([loading][loading-position="start"]) .start,
:host([loading][loading-position="end"]) .end {
  visibility: hidden;
}
:host([loading][loading-position="start"]) .spinner,
:host([loading][loading-position="end"]) .spinner {
  display: inline-flex;
  position: absolute;
  top: 50%;
  transform: translateY(-50%);
}
:host([loading][loading-position="start"]) .spinner { left: var(--_btn-padding-x); }
:host([loading][loading-position="end"]) .spinner { right: var(--_btn-padding-x); }
```
Because the spinner is `position: absolute`, it contributes **zero** inline width; the hidden slot keeps its box, so the button's intrinsic width is unchanged. This is the same mechanism `center` already uses, just edge-anchored instead of centered.
> Note: this removes the old `order: -1 / order: 99` approach (which only works for in-flow items). The absolute spinner ignores flex order, so delete those declarations.

- [ ] **Step 4: Add reduced-motion spinner pulse**

Extend the reduced-motion block:
```css
@media (prefers-reduced-motion: reduce) {
  .spinner .default-spinner { animation: tulpar-button-pulse 1.2s ease-in-out infinite; }
}
@keyframes tulpar-button-pulse { 50% { opacity: 0.4; } }
```
> Replaces the prior "slow to 1500ms" rule (still acceptable, but pulse is the spec choice). Keep the spin keyframe for the default case.

Run: `pnpm --filter @tulpar-ui/core test`
